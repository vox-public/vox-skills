## 왜 이 형식이 필요한가

vox.ai 음성 에이전트는 하나의 시스템 프롬프트 안에 대화 흐름을 정의한다. 원본 스크립트를 그대로 넣으면 LLM이 단계 경계, 전환 시점, 재시도 제한 등을 자의적으로 해석해서 예측할 수 없는 발화가 나온다. 이 규칙은 원본 스크립트를 "한 단계 = 한 목적 + 명확한 exit 조건"으로 쪼개서 LLM이 각 단계에서 할 일과 빠져나갈 조건을 정확히 파악하게 만든다.

핵심 원리:

1. **단계 안에서 할 일**(프롬프트 본문)과 **빠져나갈 조건**(전환조건)을 분리한다.
2. 전환조건에 "다음 단계"를 지정하지 않는다. exit 조건만 정의하고, 흐름 연결은 노드 순서와 LLM의 문맥 판단에 맡긴다.
3. **content에는 노드 안에서 루프하는 시나리오만 기술한다.** 전환조건이 성립하면 즉시 다음 노드로 이동하므로, 전환 후 발화는 다음 노드에서 처리한다.

## 입력

사용자가 다음 중 하나 이상을 제공한다:

- **원본 스크립트**: 통화/OB/CS 스크립트의 특정 구간(자유 텍스트). 전체 스크립트가 아닌 부분 단위로 들어온다.
- **필수 수집 항목**: 이름, 전화번호, 주소 등 반드시 수집해야 할 데이터 목록 (있으면)
- **재권유/재시도 제한**: 권유 최대 횟수, 재확인 제한 등 (있으면)
- **운영 제약**: 통화 중, 한 턴 한 질문 등 (있으면)
- **`{{...}}` 변수**: 원본 스크립트에 런타임 변수가 있으면 노드 멘트 안에 그대로 유지한다

사용자가 스크립트만 던지고 나머지를 생략하면, 스크립트에서 추론 가능한 것은 추론하고, 추론 불가능한 핵심 사항(수집 항목, 재시도 제한 등)만 짧게 질문한다. 질문은 최대 2개로 제한한다.

## 출력

마크다운 코드 스니펫(` ```md ... ``` `)으로 대화에 직접 출력한다. 파일은 만들지 않는다.

## 작업 절차

### 1단계: 스크립트 분석

원본 스크립트를 읽고 다음을 파악한다:

- 대화의 전체 목적 (세일즈/CS/예약/안내 등)
- 대화 흐름의 주요 분기점
- 각 구간에서 달성하려는 목적 (인사, 본인확인, 정보수집, 안내, 마무리 등)
- `{{...}}` 런타임 변수
- 재권유/재시도 제한이 있는 구간
- 예외 처리가 필요한 구간

### 2단계: 노드 쪼개기

스크립트를 노드 단위로 쪼갠다.

**한 노드 = 전환조건 구조가 같은 최소 대화 단위**

쪼갤 때 판단 기준 — **전환조건 구조가 핵심**:

- 후속 질문의 거절이 exit가 아니라 노드 내 루프인 경우 → 같은 노드에 포함한다. 별도 노드로 분리하면 "아니요" 시 이전 노드로 돌아가는 edge가 필요해 불필요한 복잡성이 생긴다.
  - 예: "운영시간 안내 → 문자로도 보내드릴까요?" — 거절 시 노드 안에 머물러야 하므로 하나의 노드.
- 후속 질문의 결과가 모두 exit인 경우 → 분리 가능
- 응답 처리 항목이 7개 이상이면 노드가 너무 큰 것이므로 분리 검토

### 3단계: 모드 결정 및 노드 작성

각 conversation 노드에 대해 먼저 static/dynamic 모드를 결정한 뒤, 해당 포맷으로 작성한다.

## conversation 노드 모드: static vs dynamic

conversation 노드는 두 가지 모드로 동작한다. 노드 설계 시 먼저 모드를 결정한다.

### static 모드

- `promptType: "static"`, `staticSentence` 필드 사용.
- LLM을 사용하지 않는다. `staticSentence`에 적힌 고정 멘트를 TTS로 그대로 발화한다.
- 전환조건이 성립하기 전까지 **같은 멘트를 반복**한다.
- 고객이 무슨 말을 하든 고정 멘트만 다시 말하므로, FAQ 응대나 맥락 기반 응답이 불가능하다.

적합한 상황:

- 단순 안내 후 수락/거절만 받는 노드
- 법적 고지, 수신거부 안내 등 정확한 문구가 필수인 노드
- 고객 응답에 유연한 대처가 필요 없는 노드

### dynamic 모드

- `promptType: "dynamic"`, `prompt` 필드 사용.
- `firstMessage`: 노드 진입 시 **고정으로 한 번** 발화하는 첫 멘트.
- 동작 흐름:
  1. `firstMessage`로 첫 발화 (고정 스크립트)
  2. 고객 응답 대기
  3. 전환조건 성립 → 즉시 다음 노드로 이동
  4. 전환조건 불성립 → `prompt` 기반으로 LLM이 응답 생성 → 2번으로 돌아가 반복

적합한 상황:

- 고객 질문(FAQ)에 대응해야 하는 노드
- 애매한 응답을 재확인해야 하는 노드
- 복잡한 정보 수집/선택이 필요한 노드

### 판단 기준

| 질문                                         | static          | dynamic                      |
| -------------------------------------------- | --------------- | ---------------------------- |
| 고객의 다양한 질문/반응에 대응해야 하는가?   | 아니오          | 예                           |
| 정확한 문구 그대로 발화해야 하는가?          | 예              | 아니오 (firstMessage만 고정) |
| 애매한 응답 시 재확인이 필요한가?            | 아니오          | 예                           |
| 노드 안에서 여러 턴의 대화가 오갈 수 있는가? | 아니오 (반복만) | 예                           |

## conversation 노드 출력 포맷

섹션 이름과 최상위 구분은 아래 3개로 고정한다:

- `name`
- `content`
- `transition conditions`

가독성 규칙(강제):

- `content`는 heading(###)과 ordered list(1. 2. 3.)를 적극 사용한다.
- depth는 최대 2까지만 허용한다(중첩 1단까지만). 3단 이상 중첩 금지.
- `transition conditions`는 depth 없이 1줄(또는 2문장)로만 쓴다. 중첩 리스트 금지.

### content 범위 규칙

content에는 **전환조건이 성립하지 않았을 때** 에이전트가 할 행동만 기술한다.

vox.ai flow에서 전환조건이 성립하면 현재 노드를 즉시 빠져나가고 다음 노드로 이동한다. 현재 노드에서 추가 발화를 할 기회가 없다.

**content에 넣는 것:** 노드 안에 머무르면서 대화를 이어가는 시나리오 — FAQ 응대, 재확인 질문, 애매한 응답 처리, 권유/재시도 등.

**content에 넣지 않는 것:**

- 전환조건과 1:1 대응하는 응대 멘트 (예: "거절 시: 감사합니다, 좋은 하루 되세요")
- "다음 노드로 전환" 같은 시스템 동작 설명
- 전환 후 필요한 멘트 → 다음 노드의 prompt/staticSentence/firstMessage에서 처리

### static 노드 포맷

```
## name
[노드 이름]

## content
### 목적
1. [단 하나의 목적]

### 모드
- promptType: static

### 발화 멘트
- "[고정 멘트 — TTS가 그대로 읽음. 전환조건 불성립 시 이 멘트를 반복]"

### 유의(필요 시만)
1. [재권유 제한/수집 순서 등]

## transition conditions
- [exit 상태 1]: 고객이 "[예시 발화]"처럼 [동의/의향]을 긍정으로 표현한 경우.
- [exit 상태 2]: 고객이 "[예시 발화]"처럼 [거절/보류]를 확정한 경우.
```

### dynamic 노드 포맷

```
## name
[노드 이름]

## content
### 목적
1. [단 하나의 목적 — 스코프 제한 포함]

### 모드
- promptType: dynamic
- firstMessage: "[노드 진입 시 첫 발화 — 고정 스크립트]"

### 진행 멘트
1. 기본 질문: "[firstMessage와 동일하거나 요약]"
2. 응답 처리
   1. [상황 A — 전환조건 불성립] 시: "[예시 멘트]"
   2. [상황 B — 전환조건 불성립] 시: "[예시 멘트]"
   3. 애매 시 확정 질문 1회: "[예시 멘트]"
   4. 무응답 시: "[직전 질문 또는 직전 멘트 그대로]"

### 유의(필요 시만)
1. [재권유 제한/수집 순서/스코프 제한 등]

## transition conditions
- [라벨]: 고객이 "[예시 발화1]", "[예시 발화2]" 등 [조건 설명]한 경우
- [라벨]: 고객이 "[예시 발화]"처럼 [조건 설명]한 경우
- [라벨]: 고객이 "[예시 발화]"처럼 [거절/보류/오대상]을 확정한 경우
```

### dynamic 프롬프트 작성 가이드

실제 JSON `prompt` 필드에 들어가는 프롬프트는 아래 구조 패턴을 따른다.

**기본 구조**: `# 지시` + (필요 시) `# 지식`

```
# 지시
- [핵심 행동 — 무엇을 해야 하는지]
- [스코프 제한 — 무엇을 하지 않는지]
- [재질문 형식 — 고정 문구 지정]

# 지식 (해당 시)
[참조 데이터 — 지점 목록, 상품 정보, FAQ 등]
```

**프롬프트 작성 규칙:**

1. **스코프 제한**: "~만을 알아낸다 (다른 정보를 수집하거나 제공하지 않는다)" 형태로 노드의 행동 범위를 명시적으로 한정한다.
2. **재질문 형식 지정**: `재질문은 "..."로 한다` — LLM이 자의적으로 바꾸지 않도록 고정 문구를 지정한다.
3. **지식 섹션**: 지점 목록, 운영시간, 브랜드 연락처 등 참조 데이터가 있으면 `# 지식` 섹션으로 인라인 삽입한다.
4. **검증 규칙**: 데이터 수집 노드에서 포맷 검증이 필요하면 프롬프트에 명시한다.
   - 예: `01012341234 형식의 번호가 아니라면, "번호를 확인하지 못했습니다. 정확히 다시 입력하신 후 우물정자 버튼을 눌러주세요." 라고 말한다.`
5. **다단계 안내**: 노드 내에서 안내 → 후속 질문 순서가 있으면 번호로 순서를 지정한다.
   - 예: `1. 운영시간을 안내한다. 2. 안내 이후에 문자 발송 여부를 묻는다.`

## 작성 규칙

### 턴 운영 규칙

음성 대화에서 에이전트가 한 번에 너무 많이 말하면 고객이 끊거나, 중간에 끼어들어서 대화가 꼬인다.

- 한 턴에 한 질문 또는 한 확인만 한다.
- 기본 1~2문장. 예외적으로 3문장까지 허용하는 경우:
  - 신원확인 + AI 여부 + 통화 가능 여부를 한 번에 처리할 때
  - 동의 확정 시
  - 필수 정보 2개를 동시에 수집할 때 (서로 밀접한 경우만)
- 애매한 응답이 오면 같은 단계 안에서 확정 질문 1회만 한다.
- 무응답이면 직전 질문(또는 직전 요청 멘트)을 그대로 1회 반복한다.

### 전환조건 작성 규칙

전환조건은 LLM이 "이 노드에서 빠져나가도 되는지"를 판단하는 유일한 기준이다. 잘못 쓰면 너무 빨리 넘어가거나 빠져나오지 못한다.

원칙:

- **고객 발화 기반으로만 작성한다.** 에이전트의 내부 상태나 추측이 아니라, 고객이 실제로 말한 것을 기준으로 판단한다.
- 각 노드의 exit 가능한 결과 상태를 **2~5개**로 정리한다. 대표적인 exit 상태: 성공 확정 / 대체 정보 확정 / 거절 확정 / 콜백·보류 확정 / 오대상 확정.
- **다음 단계 이름을 절대 쓰지 않는다.** "다음은 결제 안내로 넘어간다" 같은 표현 금지. 오직 "이 노드를 exit하는 경우의 수"만 나열한다.
- 예/아니오(동의/확인)처럼 **긍정 수락만 받으면 진행되는 노드**는 전환조건을 유하게 쓴다. 고객이 "네", "좋아요", "알겠습니다" 같은 짧은 긍정 응답만 해도 exit 가능하도록 포함한다. 단, 선택/정보수집처럼 구체 값이 필요한 노드는 "네"만으로는 exit하지 않게 쓴다.
- 재권유/재시도 제한이 있으면 전환조건에 "아직 사용하지 않은 경우 / 이미 사용한 경우"를 포함한다(단, 중첩 없이 1~2문장으로 풀어쓴다).
- **형식 강제:** `transition conditions`는 depth 없이 `- ` 불릿만 사용한다(하위 불릿/번호/중첩 금지).
- **문장 길이:** 각 조건은 1문장 또는 2문장으로 끝낸다.
- **논리 결합:** "조건A && 조건B 인 경우"처럼 한 줄에서 조건을 풀어쓰는 것은 허용한다(단, depth를 늘리지 않는다).

**전환조건 포맷: `라벨: 조건` 패턴**

각 전환조건은 `- [라벨]: [조건 설명]` 형태로 작성한다. 라벨은 해당 exit 상태의 이름이다.

```
- 문자 선택: 고객이 "문자", "문자로 받을게요", "링크 보내주세요" 등 문자 수신을 선택한 경우
- 담당자 연락 선택: 고객이 "담당자 연락 주세요", "상담원 연결해 주세요" 등 담당자 연락을 선택한 경우
- 거절: 고객이 "안 할게요", "필요 없어요"처럼 진행 자체를 확정 거절한 경우
```

**AI 질문 참조 패턴**: AI가 던진 질문에 대한 긍정/부정 응답이 exit 조건이면, AI 질문을 조건에 포함한다.

```
- "문자로 보내드릴까요?" 형식의 질문에 "네", "응", "어", "보내줘" 등 긍정의 답을 하는 경우
- "OO점 맞으신가요?" 형식의 질문에 "네", "맞아요" 등 맞다고 하는 경우
```

**예시 발화 나열**: 전환조건에 고객 예시 발화를 2~4개 나열한다. LLM이 유사한 발화도 매칭할 수 있도록 변형을 포함한다.

**DTMF 전환조건**: IVR 메뉴에서 DTMF 입력을 받을 때는 `[DTMF]` 접두어를 사용한다.

```
- 유저가 [DTMF] 1 을 누른 경우
- 유저가 [DTMF] 2 를 누른 경우
```

**복합 조건**: 두 조건이 동시에 성립해야 하면 `&&`로 결합한다. depth를 늘리지 않고 한 줄에 풀어쓴다.

```
- 전화번호를 3회이상 요청을 했음 && 01012341234 형식이 아닌 경우
```

### 멘트 작성 규칙

- 예시 멘트는 반드시 큰따옴표("")로 감싼다.
- 표/마크다운 테이블/괄호 연출 금지. 일반 텍스트만 사용한다.
- 이모지/이모티콘/특수 장식 문자 사용 금지.
- `{{...}}` 런타임 변수가 원본 스크립트에 있으면 멘트 안에 그대로 유지한다.
- 음성으로 자연스럽게 들리는 한국어 존댓말(높임 공손: ~실까요, ~드릴까요)로 작성한다.
- TTS가 읽을 수 없는 특수문자(\*, #, → 등)를 멘트 안에 넣지 않는다.

### 자가 점검

노드 작성이 끝나면 아래를 확인한다:

- 노드 쪼개기가 전환조건 구조에 맞는가 (후속 질문 거절이 exit가 아닌 경우 같은 노드)
- conversation 노드에 static/dynamic 모드가 명시되어 있는가
- dynamic 노드에 firstMessage가 명시되어 있는가
- dynamic 프롬프트에 스코프 제한이 있는가 (무엇을 하지 않는지)
- dynamic 프롬프트에 재질문 형식이 고정 문구로 지정되어 있는가
- content에 전환조건 성립 시의 응대 멘트가 포함되어 있지 않은가 (있으면 다음 노드로 이동)
- 전환조건이 `라벨: 조건` 형식이며, 예시 발화가 2~4개 나열되어 있는가
- 전환조건에 "다음 단계 이름"이 들어가지 않았는가
- extraction 노드의 변수 설명에 추출 소스와 `ex)` 예시 출력이 포함되어 있는가
- api 노드에 promptType(static/none)이 명시되어 있는가
- `## name` / `## content` / `## transition conditions` 3개로만 출력했는가
- `content`의 리스트 depth가 2를 초과하지 않는가
- 예시 멘트가 모두 큰따옴표로 감싸져 있는가
- 한 턴의 발화가 3문장을 초과하지 않는가
- 원본 스크립트의 `{{...}}` 변수가 빠지지 않았는가

## 노드 타입별 작성법

conversation 노드는 위 포맷을 그대로 따른다. 아래는 conversation 외 노드 타입의 작성법이다.

### extraction 노드

대화 컨텍스트에서 변수를 추출한다. 고객과 대화하지 않으며, 항상 `isSkipUserResponse: true`로 자동 전환된다.

- 변수는 대부분 **단일 변수**로 설계한다. 여러 값을 추출해야 하면 하나의 string 변수에 포맷을 지정한다.

```md
## name

[노드 이름]

## content

### 목적

1. [추출 소스]에서 [추출 대상]을 추출한다.

### 추출 변수

- [변수명] (타입): [추출 소스 명시] + [추출 대상 설명] + [포맷 규칙]
  ex) [기대 출력 예시]

## transition conditions

(자동 전환 — skipUserResponse: true. 조건 없이 다음 노드로 진행.)
```

**extraction 작성 규칙:**

1. **추출 소스 명시**: "고객이 DTMF로 입력한", "AI가 답변한", "직전 대화에서 고객이 말한" 등 추출 원천을 명확히 쓴다.
2. **`ex)` 예시 출력 필수**: 추출 결과가 어떤 형태여야 하는지 구체적 예시를 반드시 포함한다. LLM이 출력 포맷을 정확히 판단하는 핵심 단서다.
3. **포맷 검증 규칙**: 전화번호, 주문번호 등 포맷이 정해진 데이터는 형식을 명시한다.
4. **지식 인라인**: 추출 대상이 한정된 목록이면(지점 정보, 상품 목록 등) `prompt`에 참조 데이터를 인라인으로 포함한다.

**적용 예시:**

```md
## name

전화번호 추출

## content

### 목적

1. 고객이 DTMF로 입력한 전화번호를 추출한다.

### 추출 변수

- input_number (string): 고객이 DTMF로 입력한 전화번호를 추출. 010(3자리) + 8자리 숫자 형식의 총 11자리 숫자.
  ex) 01012341234

## transition conditions

(자동 전환 — skipUserResponse: true. 조건 없이 다음 노드로 진행.)
```

```md
## name

고객정보 추출

## content

### 목적

1. 직전 대화에서 고객이 말한 성함과 회원번호를 추출한다.

### 추출 변수

- customer_name (string): 고객이 말한 본인 성함
- membership_number (string): 고객이 말한 회원번호 (숫자 또는 영문+숫자 조합)

## transition conditions

(자동 전환 — skipUserResponse: true. 조건 없이 다음 노드로 진행.)
```

### condition 노드

이미 추출된 변수 값을 시스템이 기계적으로 비교하여 분기한다. 고객과 대화하지 않는다. 반드시 앞에 extraction 또는 api 노드가 있어야 한다.

```md
## name

[노드 이름]

## content

### 목적

1. [분기 판단 목적]

### 분기 조건

- {{변수명}} == "값A" → [결과 라벨]
- {{변수명}} == "값B" → [결과 라벨]
- else → [결과 라벨]

## transition conditions

(시스템 자동 분기 — 위 조건 기반. 고객 발화 아님.)
```

**적용 예시:**

```md
## name

회원등급확인

## content

### 목적

1. 회원 등급에 따라 안내 분기를 결정한다.

### 분기 조건

- {{member_grade}} == "premium" → 프리미엄혜택안내
- {{member_grade}} == "basic" → 일반혜택안내
- {{member_grade}} does_not_exist → 비회원안내

## transition conditions

(시스템 자동 분기 — 위 조건 기반. 고객 발화 아님.)
```

### api 노드

외부 HTTP API를 호출하고 응답에서 변수를 추출한다. fallback transition이 자동 생성된다.

**api 노드 promptType 옵션:**

| promptType  | 동작                              | 사용 시점                                                        |
| ----------- | --------------------------------- | ---------------------------------------------------------------- |
| `"static"`  | `staticSentence` 발화 후 API 호출 | API 호출 전 대기 멘트가 필요할 때 ("잠시만 기다려주세요")        |
| `"none"`    | 발화 없이 바로 API 호출           | 직전 노드에서 이미 대기 멘트를 발화했거나, 무음 호출이 필요할 때 |
| `"dynamic"` | LLM 생성 발화 후 API 호출         | (거의 사용하지 않음)                                             |

**"잠시만" 패킹 패턴:**

API 호출 전 대기 멘트를 처리하는 두 가지 방식:

- **방식 A**: api 노드 자체에 `promptType: "static"` + `staticSentence: "잠시만 기다려주세요."` 설정. 간단하고 노드 수가 적다.
- **방식 B**: 별도 static conversation(skipUserResponse: "잠시만 기다려주세요.") → api 노드(`promptType: "none"`). 대기 멘트와 API 호출을 분리해야 할 때 사용 (예: 대기 멘트 후 extraction을 먼저 수행해야 하는 경우).

```md
## name

[노드 이름]

## content

### 목적

1. [API 호출 목적]

### 발화 모드

- promptType: [none / static]
- staticSentence: "[대기 멘트 — static일 때만]"

### API 설정

- method: [GET/POST/PUT/DELETE/PATCH]
- url: [요청 URL — {{변수명}}으로 동적 값 삽입 가능]
- body: [요청 body — 필요 시]
- auth: [인증 방식 — 필요 시]

### 응답 변수 (해당 시)

- [변수명]: [JSONPath 표현식] — [설명]

## transition conditions

- 성공: API 응답 정상 수신 시 다음 노드로 진행.
- 실패: API 호출 실패 시 fallback edge로 진행. (fallback은 시스템 자동 생성)
```

**적용 예시:**

```md
## name

주문조회API

## content

### 목적

1. 주문번호로 외부 시스템에서 주문 상태를 조회한다.

### 발화 모드

- promptType: static
- staticSentence: "잠시만 기다려주세요."

### API 설정

- method: GET
- url: https://api.example.com/orders/{{order_number}}
- auth: Bearer (토큰은 에이전트 설정에서 주입)

### 응답 변수

- order_status: $.data.status — 주문 상태 (delivered/in_transit/cancelled)
- delivery_date: $.data.delivery_date — 배송 예정일

## transition conditions

- 성공: API 응답 정상 수신 시 다음 노드로 진행.
- 실패: API 호출 실패 시 fallback edge로 진행. (fallback은 시스템 자동 생성)
```

### endCall 노드

종료 멘트를 발화하고 통화를 종료한다. Global Node로 설정하면 어디서든 "통화 끊어주세요" 등으로 이 노드에 도달할 수 있다.

```md
## name

[노드 이름]

## content

### 목적

1. [종료 목적/상황]

### 종료 멘트

- promptType: [static/dynamic]
- 멘트: "[종료 시 발화할 멘트]"

### Global Node 설정 (해당 시)

- isGlobalNode: true
- transitionCondition: "[글로벌 진입 조건 — 예: 고객이 통화 종료를 요청한 경우]"

## transition conditions

(통화 종료 — 전환 없음.)
```

**적용 예시:**

```md
## name

종료

## content

### 목적

1. 통화를 마무리하고 종료한다.

### 종료 멘트

- promptType: dynamic
- 멘트: "네, 상담 도와드려서 감사합니다. 좋은 하루 되세요."

### Global Node 설정

- isGlobalNode: true
- transitionCondition: "고객이 '끊어주세요', '그만할게요', '됐어요' 등 통화 종료 의사를 표현한 경우"

## transition conditions

(통화 종료 — 전환 없음.)
```

## Global Node 설계 가이드

콜센터 스크립트에 거의 항상 존재하는 "언제든 상담원 연결", "언제든 통화 종료" 같은 글로벌 예외를 flow에 반영한다.

**Global Node 후보 판별:**

- 스크립트에 "언제든지", "어떤 단계에서든" 같은 표현이 있으면 Global Node 후보
- 대표적 패턴: 통화 종료 요청, 상담원 연결 요청, 불만/욕설 대응

**설계 규칙:**

1. Global Node로 설정 가능한 타입: conversation, endCall
2. `isGlobalNode: true` + `transitionCondition`을 반드시 함께 설정한다
3. `transitionCondition`은 고객 발화 기반으로 작성한다 — "고객이 ~라고 말한 경우"
4. Global Node는 최소화한다 — 2~3개 이내. 너무 많으면 전환 충돌 위험

**일반적인 Global Node 구성:**

| 글로벌 예외       | 노드 타입                   | transitionCondition 예시                             |
| ----------------- | --------------------------- | ---------------------------------------------------- |
| 통화 종료         | endCall                     | "고객이 통화 종료를 요청한 경우"                     |
| 상담원 연결       | conversation → transferCall | "고객이 상담원 연결을 요청한 경우"                   |
| 불만 에스컬레이션 | conversation → transferCall | "고객이 강한 불만을 표현하거나 책임자를 요구한 경우" |

## 적용 예시

### static 노드 예시

수락/거절만 받으면 되는 단순 안내. LLM 불필요.

```md
## name

수신거부 안내

## content

### 목적

1. 수신거부 방법을 안내하고 통화 계속 여부를 확인한다.

### 모드

- promptType: static

### 발화 멘트

- "본 전화는 무료 수신 거부 안내를 포함하고 있습니다. 수신을 원치 않으시면 아니오라고 말씀해주세요. 안내를 계속 들으시겠습니까?"

## transition conditions

- 계속 수신 확정: 고객이 "네", "들을게요"처럼 안내 계속을 수락한 경우.
- 수신거부 확정: 고객이 "아니오", "안 들을래요", "거부할게요"처럼 수신거부 의사를 표현한 경우.
```

### dynamic 노드 예시

고객 선택, FAQ, 권유가 필요한 복합 대화. `firstMessage`로 첫 발화를 고정하고, 이후 LLM이 대화를 이어간다. content에는 **전환조건 불성립 시** 노드 안에서 루프하는 시나리오만 기술한다.

```md
## name

결제방법 안내

## content

### 목적

1. 고객이 결제 방식(전액 또는 예약금)을 선택하도록 돕는다.

### 모드

- promptType: dynamic
- firstMessage: "결제방법 안내 도와드리겠습니다. 전액 결제와 예약금 결제 중 어떤 방식으로 안내 도와드릴까요?"

### 노드 내 대화 처리

1. 선택을 못 하는 경우 1차 전액결제 권유: "방송 중 안내된 결제 혜택은 전액 결제에서만 제공되고 있습니다. 전액 결제로 안내 도와드릴까요?"
2. 전액이 부담스럽다고 하면 2차 예약금 권유: "네, 부담되실 수 있어요. 그럼 예약금 결제 안내로 도와드릴까요?"
3. 회원가입 관련 질문 시: "네, 전액 결제는 회원가입 후 진행이 필요하고요, 예약금 결제도 예약금 결제 후 회원가입이 필요합니다. 원하시는 결제 방식부터 선택해주실까요?"
4. 애매 시 확정 질문 1회: "정확히 확인드리려고요, 전액 결제와 예약금 결제 중 어느 쪽으로 안내 도와드릴까요?"
5. 무응답 시 반복 1회: "[직전 질문 또는 직전 멘트 그대로]"
6. 유의
   1. 전액결제 권유는 1회, 예약금 전환 권유는 1회까지만 한다.
```
