# Conversation node markdown

conversation 노드는 고객 발화를 듣고 LLM 또는 고정 멘트로 응답하면서 exit 조건을 판단하는 대화 단계다. 이 문서는 **설계 markdown** 작성법만 다룬다. JSON field 는 `flow-schema/flow-data` schema endpoint 결과를 따른다.

> **용어 안내**: 대시보드 / 사내 문서에서 흔히 "static / dynamic" 이라고 부르는 두 모드는 schema 표면에서 `message.mode = "static" | "generated"` 로 노출된다. 이 문서에서는 schema 명칭인 **static / generated** 를 일관되게 쓰지만, 의미는 동일하다 (dynamic = generated). dynamic(generated) 모드를 쓸 때는 `first_message` 를 함께 명시한다.

## Mode selection

| 상황 | 권장 mode |
|---|---|
| 정확한 문구를 그대로 말해야 함 | static |
| 고객 질문, 애매한 응답, 재확인에 대응해야 함 | generated |
| 단순 안내 후 사용자 응답 없이 넘어가야 함 | static + 사용자 응답 대기 없음 의도 명시 |

static 은 같은 고정 멘트를 반복할 수 있다. FAQ 대응이나 맥락 기반 응답이 필요하면 generated 로 둔다.

generated 는 첫 발화는 고정하고, 이후에는 노드 안에서만 처리할 시나리오를 제한한다. "무엇을 하지 말아야 하는지"를 함께 쓴다.

## Static format

```md
## name
[노드 이름]

## content
### 목적
1. [단 하나의 목적]

### 모드
- message mode: static

### 발화 멘트
- "[고정 멘트]"

### 유의
1. [필요할 때만: 반복 제한, 수신거부 고지 등]

## transition conditions
- [exit 상태 1]: 고객이 "[예시]"처럼 [조건]을 표현한 경우.
- [exit 상태 2]: 고객이 "[예시]"처럼 [조건]을 표현한 경우.
```

## Generated format

```md
## name
[노드 이름]

## content
### 목적
1. [단 하나의 목적과 스코프 제한]

### 모드
- message mode: generated
- first_message: "[노드 진입 시 첫 발화]"

### 노드 내 대화 처리
1. 기본 질문: "[first_message와 동일하거나 짧은 재질문]"
2. [상황 A, 전환조건 불성립] 시: "[예시 멘트]"
3. [상황 B, 전환조건 불성립] 시: "[예시 멘트]"
4. 애매한 응답 시: "[확정 질문]"
5. 무응답 시: "[반복 또는 짧은 재질문]"

### 유의
1. [재권유 제한, 수집 순서, 금지할 응대]

## transition conditions
- [exit 상태 1]: 고객이 "[예시]"처럼 [조건]을 표현한 경우.
- [exit 상태 2]: 고객이 "[예시]"처럼 [조건]을 표현한 경우.
```

## 발화 멘트와 변수 결합

발화 멘트(`first_message`, static 발화, generated 의 노드 안 재질문)와 `{{variable}}` 의 결합부에서 가장 자주 깨지는 곳은 **어휘 중복** 이다. 멘트의 자연어 표현이 변수 값과 같은 단어를 포함하면 TTS 출력에서 의도하지 않은 중복이 발생한다.

대표적 함정:

- 변수 값: `{{appointment_datetime}} = "내일 오후 2시"`
  멘트: `"{{customer_name}}님, 내일 {{appointment_datetime}}에 예약이 있습니다."`
  실제 발화: `"홍길동님, 내일 내일 오후 2시에 예약이 있습니다."` ← "내일" 중복.
- 변수 값: `{{location}} = "강남점"`
  멘트: `"강남에 있는 {{location}}으로 오시면 됩니다."` ← "강남" 중복.
- 변수 값: `{{customer_name}} = "홍길동 고객"`
  멘트: `"{{customer_name}}님, 안녕하세요."` ← "고객님" 중복 가능.

작성 규칙:

1. **변수 값을 멘트에 미리 대입해 본다** — `{{customer_name}}` 가 통화 시작 전 어떻게 주입될지, `{{appointment_datetime}}` 이 어떤 표현으로 추출될지 구체적인 예시 값을 머릿속에 두고 멘트를 쓴다.
2. **시간/장소/이름 어휘를 변수와 멘트 양쪽에 동시 박지 않는다** — 시간 표현(`내일`, `오늘`, `이번 주`)이 변수 값 안에 포함될 가능성이 있으면 멘트의 시간 어휘는 빼고 변수에만 둔다.
3. **변수 값 끝에 조사 / 단위 / 호칭** 을 변수 안에 넣지 않는다 — `{{customer_name}}` 에 "홍길동" 만, `{{location}}` 에 "강남점" 만. 조사·호칭("님", "에", "으로") 은 멘트에 둔다.
4. **멘트 작성 후 가장 빈번할 변수 값 1-2 개를 직접 대입해 출력 문장을 읽어본다** — 부자연스러운 중복/반복이 보이면 멘트를 다시 쓴다.
5. agent.data.presetDynamicVariables 에서 사전 주입되는 변수는 의미상 명사만 담는다 ("내일 오후 2시" 같은 자연어 그대로 주입은 가능하지만 멘트에 시간 어휘 박힐 가능성이 같이 커짐).

이 규칙은 static / generated 모드 모두에 적용된다. generated 모드의 first_message 가 가장 자주 새는 곳이지만, static 의 고정 멘트도 변수 값 폭에 따라 같은 함정에 빠질 수 있다.

## Content boundary

`content`에는 현재 노드 안에서 계속할 행동만 쓴다.

넣는다:
- FAQ 응대
- 재확인 질문
- 애매한 응답 처리
- 노드 안 재권유/재시도
- 무응답 처리

넣지 않는다:
- 전환조건 성립 후의 응대 멘트
- "다음 노드로 이동" 같은 시스템 동작 설명
- 다음 노드에서 말해야 할 안내

## Transition conditions

- 다음 노드 이름을 쓰지 않는다. exit 조건만 쓴다.
- 예시 발화는 2-4개면 충분하다.
- 전환조건은 한 줄로 쓴다. 하위 불릿을 만들지 않는다.
- 고객 발화 기반 조건과 변수 기반 조건을 섞지 않는다. 변수 기반 분기는 condition 노드로 보낸다.
- "동의/거절" 같은 자연어 판단은 보통 conversation out-edge 조건이다.

### ai condition 자연어는 broad 하게

ai condition 의 prompt 가 좁으면 (예: "본인이 아니다 / 잘못 거셨다" 만 적힘) 실제 거절·보류 발화의 다양한 표현 (`"스팸이냐"`, `"누구야"`, `"바빠"`, `"안 받을래요"`) 을 매칭하지 못한다. 결과적으로 LLM 이 노드 안에 갇혀 자체 마무리 발화를 하거나, 글로벌 노드로 빠져 종료된다.

작성 규칙:

1. **상위 의도 + 동의어 묶기** 형태로 쓴다 — "고객이 통화를 거절하거나, 본인이 아니라고 하거나, 바빠서 못 받겠다고 하는 등 통화 의사가 없는 경우" 처럼 상위 의도에 동의어 표현 다수를 묶어둔다.
2. **예시 발화 2-3 개** 를 prompt 에 직접 포함하면 LLM 매칭이 안정적이다 — `"고객이 '스팸이냐', '바빠', '안 들을래요' 같이 통화 의사 없음을 표현한 경우"`.
3. **거절 / 보류 / 화제 전환** 같이 자주 나오는 카테고리는 고정 어휘 묶음을 만들어 두고 재사용한다:
   - 통화 거절: `"스팸이냐"`, `"바빠"`, `"안 들을래요"`, `"누구세요"`(부정 톤), `"왜 전화했어"`
   - 보류: `"나중에 다시"`, `"좀 생각해보고"`, `"지금은 곤란"`, `"아직 잘 모르겠"`
   - 화제 전환: `"그게 아니라"`, `"다른 거 물어봐도 돼요"`, 새 질문 시작
4. **부정 케이스 (어떤 발화일 때 매칭 안 되는지) 도 한 줄 추가** 하면 LLM 이 더 정확히 판단한다 — `"단, '잠시만요'처럼 일시 보류만 표현한 경우는 제외"`.
5. **condition 별 mutually exclusive** 하게 작성한다 — 같은 발화가 두 condition 에 동시 매칭되면 우선순위가 모호해진다.

prompt 길이는 한두 문장 안쪽이 권장이지만, broad 표현을 위해 동의어 묶음이 들어가면 3-4 문장까지도 허용된다. 모호하게 짧게 쓰는 것보다 명확하게 길게 쓰는 게 매칭률이 높다.

## Prompt guardrails

generated 노드에는 아래를 짧게 포함한다.

- 이 노드의 목표.
- 이 노드에서 수집하거나 확정할 것.
- 이 노드에서 하지 말아야 할 것.
- 고객이 질문했을 때 답할 수 있는 범위.
- 재질문/재권유 최대 횟수.
- 발화 톤 의도 (제안 / 확인 / 안내 / 확정).

### 발화 톤 제어 (제안 vs 확정)

flow 가 여러 노드로 의사결정을 단계별로 진행할 때, 각 노드의 **톤 의도** 를 명시하지 않으면 LLM 이 한 노드에서 "확정 멘트" 까지 발화해버려 다음 노드의 재확인이 어색해진다. 예: 변경 시간을 확정하기 전 단계에서 LLM 이 `"이번 주 금요일 오후 2시로 변경해 드리겠습니다"` (확정톤) 발화 후, 다음 노드가 다시 `"이대로 확정해도 괜찮으실까요?"` 묻는 이중 확정.

규칙:

1. **노드 의도 = 발화 톤** 으로 prompt 에 명시한다.
   - 제안 톤: `"이 노드의 목표는 옵션을 제안하는 것이다. 확정 표현 ('변경해드리겠습니다', '예약 처리하겠습니다') 은 사용하지 말고, 제안 표현 ('변경 도와드릴까요', '이 시간으로 안내드릴까요') 만 쓴다."`
   - 확인 톤: `"고객이 선택한 값을 다시 한 번 복기하고 동의 / 부정 응답을 받는 것이 목표. 새 정보는 추가하지 않는다."`
   - 안내 톤: `"정해진 사실을 일방적으로 안내한다. 추가 질문은 받지 않는다."`
   - 확정 톤: `"이 노드는 확정 직후 endCall 또는 후속 처리로 넘어간다. 처리 완료를 명확히 알린다."`
2. **확정 멘트는 항상 다음 노드에서** 한다. 같은 노드 안에서 제안 → 확정을 동시에 표현하지 않는다.
3. 노드별로 어떤 표현을 사용할지 / 사용 금지할지 prompt 안에 짧게 박아둔다 — 자주 새는 표현은 명시적으로 차단 (`"~해드리겠습니다"`, `"처리되었습니다"` 류).
4. 제안 → 확정 은 보통 두 개의 conversation 노드로 분리된다. extraction 노드를 사이에 두고 변수에 선택값을 추출한 뒤 다음 conversation 에서 확정 멘트.

### 재시도 횟수 통제

generated 노드는 prompt 만으로 재질문 횟수를 통제하지 않으면 LLM 이 같은 질문을 4-5 번 반복할 수 있다. 본인확인 / 의향확인 / 정보수집 노드에서 자주 발생.

규칙:

1. **재질문 최대 횟수를 prompt 에 숫자로 박는다** — `"같은 질문을 최대 2회까지만 시도한다. 2회 시도 후에도 응답이 모호하면 다음 단계로 넘어간다."`
2. **N 회 시도 후의 fallback path 를 transition conditions 에 명시** 한다 — 한 노드의 transition 에 "응답 모호 / 무응답이 N회 이상 지속" 같은 exit 조건을 두고, 그 edge 를 fallback / 안내 노드로 연결한다.
3. `noinput` / `nomatch` 같은 패턴이 필요하면 `agent.data.callSettings` 의 무응답 처리와 노드 prompt 가 일관되도록 맞춘다.
4. 재시도가 1 회면 prompt 에서 `"한 번만 정중히 다시 묻고 그 다음에는 다음 단계로 넘어간다"` 식으로 명시한다 — 숫자가 없으면 LLM 이 자기 판단으로 늘리는 경향이 있다.

## Quick example

```md
## name
결제방법 안내

## content
### 목적
1. 고객이 전액 결제와 예약금 결제 중 하나를 선택하도록 돕는다. 이 단계에서는 결제 수단 입력을 받지 않는다.

### 모드
- message mode: generated
- first_message: "결제방법 안내 도와드리겠습니다. 전액 결제와 예약금 결제 중 어떤 방식으로 안내 도와드릴까요?"

### 노드 내 대화 처리
1. 선택을 못 하는 경우: "방송 중 안내된 혜택은 전액 결제에서만 제공되고 있습니다. 전액 결제로 안내 도와드릴까요?"
2. 전액이 부담스럽다고 하면: "부담되실 수 있어요. 그러면 예약금 결제로 안내 도와드릴까요?"
3. 애매한 응답 시: "정확히 확인드리려고요. 전액 결제와 예약금 결제 중 어느 쪽으로 안내 도와드릴까요?"
4. 무응답 시: "결제 방식 선택이 필요합니다. 전액 결제와 예약금 결제 중 어느 쪽으로 안내 도와드릴까요?"

### 유의
1. 전액결제 권유는 1회, 예약금 전환 권유는 1회까지만 한다.

## transition conditions
- 전액결제 선택 확정: 고객이 "전액으로 할게요", "전액 결제요"처럼 전액결제를 명확히 선택한 경우.
- 예약금결제 선택 확정: 고객이 "예약금으로 할게요", "예약금 결제요"처럼 예약금결제를 명확히 선택한 경우.
```
