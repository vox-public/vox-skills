# IVR Navigation Best Practices

`send_dtmf`는 단순히 "번호를 누르는 도구"가 아니라, 자동응답 메뉴 안에서 올바른 사람 또는 올바른 부서까지 도달하기 위한 탐색 전략의 일부입니다.

이 문서는 아래 두 문서의 공통 원칙을 vox.ai single prompt / skill 환경에 맞게 재구성한 실무 가이드입니다.

- Retell: [Press digit](https://docs.retellai.com/build/single-multi-prompt/press-digit)
- Vapi: [IVR Navigation](https://docs.vapi.ai/ivr-navigation)

## 핵심 원칙

- `send_dtmf`의 `description`만 길게 써서는 품질이 충분히 올라가지 않습니다.
- 실제 성공률은 아래 4가지를 프롬프트에 분리해 넣었는지에 달려 있습니다.
  - 목표: 어느 부서, 어느 사람에게 도달해야 하는지
  - 선택 기준: 어떤 메뉴를 우선 선택하고 어떤 메뉴를 피해야 하는지
  - 상호작용 규칙: 언제 기다리고, 언제 누르고, 언제 말할지
  - 실패 처리: 루프, 무응답, 잘못 연결, 음성 fallback 시 어떻게 처리할지
- IVR에서는 사람 대화보다 보수적으로 동작하는 편이 안전합니다.
- 자동 안내와 겹쳐서 말하거나, 메뉴가 끝나기 전에 숫자를 누르는 행동은 기본적으로 피합니다.

## 어떤 정보를 먼저 정해야 하나

IVR 설계를 시작하기 전에 아래를 먼저 확정합니다.

1. 최종 목표
   - 예: 시설 담당자 연결, 사람 상담원 연결, 예약 부서 연결
2. 선호 메뉴
   - 예: 시설, 전기, 설비, 영선, 상담원
3. 회피 메뉴
   - 예: 회계, 경리, 택배, 방문예약
4. 애매할 때의 우선순위
   - 예: 정확한 일치가 없으면 사람 연결에 가장 가까운 메뉴를 선택
5. 재시도 정책
   - 예: 같은 메뉴는 1회만 재시도, 이후 음성 fallback

## `send_dtmf` description 설계 원칙

`description`에는 아래만 짧고 강하게 넣는 것이 좋습니다.

- 이 도구의 목적
- 우선 연결 대상
- 회피해야 할 메뉴 유형
- 애매할 때의 선택 기준
- 재시도 또는 fallback의 상한

`description`에 넣지 말아야 하는 것:

- 장문의 시나리오 설명
- 모든 예외 상황의 세부 절차
- 고객에게 들려줄 발화 예시
- 프롬프트 안에서 더 잘 다룰 수 있는 세밀한 분기

권장 예시:

```text
이 도구는 IVR 메뉴에서 목표 부서나 실제 응대 가능한 사람에게 연결될 때 사용합니다. 메뉴 안내가 끝난 뒤에만 사용하고, 정확한 일치가 없으면 목표와 가장 가까운 부서 또는 상담원, 대표, 기타 문의처럼 사람 연결에 가까운 옵션을 선택합니다. 회계, 경리, 관리비, 임대, 분양, 택배, 방문예약처럼 목적과 무관한 메뉴는 피합니다. 같은 메뉴가 반복되면 1회만 재시도하고, 이후에는 음성 입력 또는 사람 연결 fallback을 우선합니다.
```

## Prompt에 반드시 분리해서 넣을 블록

### 1. IVR Navigation Goal

에이전트가 어디로 도달해야 하는지를 한 줄로 분명히 둡니다.

예시:

```md
## IVR Goal

목표는 시설 또는 전기 담당자, 없으면 실제 응대 가능한 직원에게 연결되는 것이다.
```

### 2. Preferred / Avoid Keywords

비슷한 메뉴가 많을수록 중요합니다.

예시:

```md
## IVR Navigation

Preferred keywords:
- 시설
- 전기
- 설비
- 영선
- 기전
- 방재
- 당직
- 상담원
- 대표
- 기타 문의

Avoid keywords:
- 회계
- 경리
- 관리비
- 임대
- 분양
- 택배
- 방문예약
```

### 3. IVR Interaction Rules

실제 실행 규칙은 이 블록에서 제어합니다.

예시:

```md
## IVR Interaction Rules

1. 자동 안내가 시작되면 끝까지 듣고 먼저 말하지 않는다.
2. IVR이 번호 입력을 명시할 때만 `send_dtmf`를 사용한다.
3. IVR이 부서명을 말하라고 하면 짧게 목표 부서명만 말한다.
4. 사람 연결 또는 보류음 전환 후에는 더 이상 IVR에 말하거나 DTMF를 보내지 않는다.
5. 같은 메뉴가 반복되면 1회만 재시도한다.
6. 재시도 후에도 실패하면 음성 입력 또는 상담원 연결 fallback을 시도한다.
```

## Skill 설계 원칙

skill 기반 에이전트에서는 `description`보다 `skill_rules`와 `skill_flow`가 실제 동작을 더 강하게 제어합니다.

### `skill_rules`에 넣을 내용

`skill_rules`에는 전역 원칙과 금지사항을 넣습니다.

권장 항목:

- 자동 안내 중 침묵 유지
- 숫자 입력을 명시했을 때만 `@tool:send_dtmf` 사용
- 우선 연결 대상
- 회피 메뉴
- 재시도 상한
- 음성 fallback 허용 조건
- 사람 연결 후 추가 DTMF 금지

예시:

```xml
<skill_rules>
## send_dtmf 사용 원칙

- MUST: 자동 안내멘트가 시작되면 말을 하지 않고 끝까지 듣는다.
- MUST: IVR이 숫자 입력을 요구할 때만 `@tool:send_dtmf`를 사용한다.
- MUST: 목표와 직접 연결되는 메뉴가 있으면 그 메뉴를 우선 선택한다.
- MUST: 정확한 일치가 없으면 상담원, 대표, 기타 문의처럼 사람 연결에 가장 가까운 메뉴를 선택한다.
- MUST NOT: 목적과 무관한 메뉴를 선택하지 않는다.
- MUST: 같은 메뉴에 대한 재시도는 1회까지만 허용한다.
- MUST: 재시도 후에도 실패하면 음성 입력 또는 상담원 연결 fallback을 검토한다.
- MUST: 보류음 또는 사람 연결 이후에는 `@tool:send_dtmf`를 다시 호출하지 않는다.
</skill_rules>
```

### `skill_flow`에 넣을 내용

`skill_flow`에는 실제 순서를 넣습니다. 특히 아래 상태 전환을 분명히 분리하는 것이 중요합니다.

- 자동 안내 감지
- 숫자 입력 요구
- 음성 입력 요구
- 보류음/연결음
- 사람 응답
- 루프 또는 실패

예시:

```xml
<skill_flow>
## 연결 대기

1. 상대방의 첫 응답을 기다린다
  1. 자동 안내멘트 패턴 감지 시 `#block:IVR 탐색`으로 이동한다
  2. 사람 응답 패턴 감지 시 다음 블록으로 이동한다

## IVR 탐색

1. 자동 안내가 시작되면 메뉴가 끝날 때까지 기다린다
2. 번호 입력을 요구하면 메뉴를 비교한다
  1. 목표와 직접 연결되는 메뉴가 있으면 `@tool:send_dtmf`를 호출한다
  2. 직접 연결 메뉴가 없고 상담원 또는 대표 연결이 있으면 `@tool:send_dtmf`를 호출한다
  3. 목적과 무관한 메뉴만 있으면 다음 안내를 더 듣는다
3. 부서명을 말하라고 하면 짧게 목표 부서명만 말한다
4. 같은 메뉴가 반복되면 같은 의도로 1회만 재시도한다
5. 보류음 또는 담당자 응답이 나오면 더 이상 `@tool:send_dtmf`를 호출하지 않는다
</skill_flow>
```

## single prompt에서의 설계 원칙

single prompt 에이전트에서도 구조는 동일합니다. 다만 XML 태그 대신 Markdown 섹션으로 분리합니다.

권장 섹션:

- `# 도구`
- `## IVR Goal`
- `## IVR Navigation`
- `## IVR Interaction Rules`
- `## Failure Handling`

## Retry / Fallback 설계

재시도는 명확하게 상한을 둬야 합니다.

- 같은 입력 무한 반복 금지
- 동일 의도 재시도는 보통 1회까지만 허용
- 가능하면 재시도 시 더 느리게 입력
- 여전히 실패하면 음성 fallback
- 음성 fallback도 실패하면 사람 연결 또는 종료 정책으로 이동

권장 순서:

1. 메뉴를 끝까지 듣는다
2. 명시된 번호가 있으면 `send_dtmf`
3. 실패 시 1회 재시도
4. 음성 입력 가능 시 짧은 부서명 발화
5. 그래도 실패하면 상담원/대표/기타 문의 fallback
6. 여전히 실패하면 종료 또는 후속 처리

## 운영 중 꼭 수집할 항목

IVR 튜닝은 로그 없이는 잘 되지 않습니다. 가능하면 아래 항목을 post-call 또는 운영 로그에 남깁니다.

- `hit_ivr`
- `reached_human`
- `ivr_type`
- `ivr_path`
- `ivr_steps_count`
- `ivr_retries_count`
- `ivr_loop`
- `ivr_failure_reason`

이 정보가 있어야 "어느 메뉴에서", "몇 단계째", "어떤 fallback에서" 막히는지 볼 수 있습니다.

## 자주 나오는 안티패턴

- 자동 안내 중간에 "네", "여보세요"라고 끼어드는 설계
- 메뉴가 끝나기 전에 숫자를 누르는 설계
- 목적과 가까워 보인다는 이유만으로 첫 비슷한 메뉴를 무조건 선택하는 설계
- 재시도 상한 없이 같은 번호를 반복 입력하는 설계
- 보류음과 IVR을 같은 상태로 취급하는 설계
- `description`만 길게 쓰고 정작 prompt/skill에는 규칙을 두지 않는 설계

## 관리사무소용 추천 우선순위

전기차 충전기, 차단기 리셋, 시설 협조 요청처럼 관리사무소에 거는 콜에서는 아래 우선순위를 권장합니다.

### Preferred

- 시설
- 전기
- 설비
- 영선
- 기전
- 방재
- 당직
- 관리실

### Fallback

- 상담원
- 대표
- 기타 문의
- 0번

### Avoid

- 회계
- 경리
- 관리비
- 분양
- 임대
- 택배
- 방문예약

## 최종 요약

- `send_dtmf`는 도구 자체보다 탐색 규칙이 중요합니다.
- `description`은 짧고 명확하게 씁니다.
- 실제 품질은 `IVR Goal`, `Preferred/Avoid`, `Interaction Rules`, `Retry/Fallback`에 달려 있습니다.
- skill에서는 `skill_rules`와 `skill_flow`에 IVR 전용 블록을 분리해 넣는 것이 가장 안정적입니다.
