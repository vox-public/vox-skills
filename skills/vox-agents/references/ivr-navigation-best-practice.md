# IVR Navigation Best Practices

`send_dtmf`는 단순히 "번호를 누르는 도구"가 아니라, 자동응답 메뉴 안에서 올바른 사람 또는 올바른 부서까지 도달하기 위한 탐색 전략의 일부입니다.

이 문서는 IVR 탐색 전략을 vox.ai 에이전트 환경에 맞게 정리한 실무 가이드입니다.

## 핵심 원칙

- `send_dtmf`의 `description`만 길게 써서는 품질이 충분히 올라가지 않습니다.
- 실제 성공률은 아래 4가지를 프롬프트에 분리해 넣었는지에 달려 있습니다.
  - 목표: 어느 부서, 어느 사람에게 도달해야 하는지
  - 선택 기준: 어떤 메뉴를 우선 선택하고 어떤 메뉴를 피해야 하는지
  - 상호작용 규칙: 언제 기다리고, 언제 누르고, 언제 말할지
  - 실패 처리: 루프, 무응답, 잘못 연결, 음성 fallback 시 어떻게 처리할지
- IVR에서는 사람 대화보다 보수적으로 동작하는 편이 안전합니다.
- 자동 안내와 겹쳐서 말하거나, 메뉴가 끝나기 전에 숫자를 누르는 행동은 기본적으로 피합니다.

## 어떤 정보를 먼저 정해야 하나

IVR 설계를 시작하기 전에 아래를 먼저 확정합니다.

1. 최종 목표
   - 예: 기술지원팀 연결, 사람 상담원 연결, 예약 부서 연결
2. 선호 메뉴
   - 예: 목표 부서와 직접 관련된 키워드
3. 회피 메뉴
   - 예: 목적과 무관한 부서(회계, 경리 등)
4. 애매할 때의 우선순위
   - 예: 정확한 일치가 없으면 사람 연결에 가장 가까운 메뉴를 선택
5. 재시도 정책
   - 예: 같은 메뉴는 1회만 재시도, 이후 음성 fallback

## `send_dtmf` description 설계 원칙

`description`에는 아래만 짧고 강하게 넣는 것이 좋습니다.

- 이 도구의 목적
- 우선 연결 대상
- 회피해야 할 메뉴 유형
- 애매할 때의 선택 기준
- 재시도 또는 fallback의 상한

`description`에 넣지 말아야 하는 것:

- 장문의 시나리오 설명
- 모든 예외 상황의 세부 절차
- 고객에게 들려줄 발화 예시
- 프롬프트 안에서 더 잘 다룰 수 있는 세밀한 분기

권장 예시:

```text
이 도구는 IVR 메뉴에서 목표 부서나 실제 응대 가능한 사람에게 연결될 때 사용합니다. 메뉴 안내가 끝난 뒤에만 사용하고, 정확한 일치가 없으면 목표와 가장 가까운 부서 또는 상담원 연결에 가까운 옵션을 선택합니다. 목적과 무관한 메뉴는 피합니다. 같은 메뉴가 반복되면 1회만 재시도하고, 이후에는 음성 입력 또는 사람 연결 fallback을 우선합니다.
```

## Prompt에 반드시 분리해서 넣을 블록

### 1. IVR Navigation Goal

에이전트가 어디로 도달해야 하는지를 한 줄로 분명히 둡니다.

예시:

```md
## IVR Goal

목표는 [[목표 부서/담당자]], 없으면 실제 응대 가능한 직원에게 연결되는 것이다.
```

### 2. Preferred / Avoid Keywords

비슷한 메뉴가 많을수록 중요합니다.

예시:

```md
## IVR Navigation

Preferred keywords:
- [[목표와 직접 관련된 부서/키워드]]
- 상담원
- 대표
- 기타 문의

Avoid keywords:
- [[목적과 무관한 부서/키워드]]
```

`[[...]]`는 에이전트의 실제 업무에 맞게 채운다. 키워드가 많을수록 IVR 메뉴 매칭 정확도가 올라간다.

### 3. IVR Interaction Rules

실제 실행 규칙은 이 블록에서 제어합니다.

예시:

```md
## IVR Interaction Rules

1. 자동 안내가 시작되면 끝까지 듣고 먼저 말하지 않는다.
2. IVR이 번호 입력을 명시할 때만 `send_dtmf`를 사용한다.
3. IVR이 부서명을 말하라고 하면 짧게 목표 부서명만 말한다.
4. 사람 연결 또는 보류음 전환 후에는 더 이상 IVR에 말하거나 DTMF를 보내지 않는다.
5. 같은 메뉴가 반복되면 1회만 재시도한다.
6. 재시도 후에도 실패하면 음성 입력 또는 상담원 연결 fallback을 시도한다.
```

## Retry / Fallback 설계

재시도는 명확하게 상한을 둬야 합니다.

- 같은 입력 무한 반복 금지
- 동일 의도 재시도는 보통 1회까지만 허용
- 가능하면 재시도 시 더 느리게 입력
- 여전히 실패하면 음성 fallback
- 음성 fallback도 실패하면 사람 연결 또는 종료 정책으로 이동

권장 순서:

1. 메뉴를 끝까지 듣는다
2. 명시된 번호가 있으면 `send_dtmf`
3. 실패 시 1회 재시도
4. 음성 입력 가능 시 짧은 부서명 발화
5. 그래도 실패하면 상담원/대표/기타 문의 fallback
6. 여전히 실패하면 종료 또는 후속 처리

## 운영 중 꼭 수집할 항목

IVR 튜닝은 로그 없이는 잘 되지 않습니다. 가능하면 아래 항목을 post-call 또는 운영 로그에 남깁니다.

- `hit_ivr`
- `reached_human`
- `ivr_type`
- `ivr_path`
- `ivr_steps_count`
- `ivr_retries_count`
- `ivr_loop`
- `ivr_failure_reason`

이 정보가 있어야 "어느 메뉴에서", "몇 단계째", "어떤 fallback에서" 막히는지 볼 수 있습니다.

## 자주 나오는 안티패턴

- 자동 안내 중간에 "네", "여보세요"라고 끼어드는 설계
- 메뉴가 끝나기 전에 숫자를 누르는 설계
- 목적과 가까워 보인다는 이유만으로 첫 비슷한 메뉴를 무조건 선택하는 설계
- 재시도 상한 없이 같은 번호를 반복 입력하는 설계
- 보류음과 IVR을 같은 상태로 취급하는 설계
- `description`만 길게 쓰고 정작 프롬프트에는 규칙을 두지 않는 설계

## 최종 요약

- `send_dtmf`는 도구 자체보다 탐색 규칙이 중요합니다.
- `description`은 짧고 명확하게 씁니다.
- 실제 품질은 `IVR Goal`, `Preferred/Avoid`, `Interaction Rules`, `Retry/Fallback`에 달려 있습니다.
