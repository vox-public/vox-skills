# 음성 AI 플레이북

한국어 음성 에이전트의 system prompt를 설계할 때 자주 필요한 “원칙 + 패턴 + 금지사항”을 한 문서로 묶었습니다. prompt agent와 flow agent 모두에 적용됩니다.

## 빠른 탐색

- 워크플로우: 아래 `워크플로우 체크리스트`
- 초안 만들기: `템플릿 기반 초안`
- 핵심 규칙 모음: `필수 규칙`
- 정규화: `문자 정규화`
- 점검: `배포 전 체크리스트`
- 디버깅(원인 진단): `references/voice-ai-prompt-diagnosis.md`
- 리팩터링(개선안 생성): `references/voice-ai-prompt-revision.md`

## 워크플로우 체크리스트

아래를 답변에 그대로 복사해서 진행 상태를 체크하세요.

```
Task Progress:
- [ ] Step 1: 입력 질문(업무/컨텍스트/성공정의/도구/데이터/변수/대화 형태) 수집
- [ ] Step 2: 변수(`{{...}}`) 목록 및 “미주입 시 대체 표현” 규칙 확정
- [ ] Step 3 (optional): 말투/필러 설정(기본값 사용 가능) 및 금지 상황 확정
- [ ] Step 4: 템플릿을 채워 system prompt 초안 작성
- [ ] Step 5: 도구 사용/실패 처리/무음 액션 규칙 추가
- [ ] Step 6: 정규화(Spoken/Written) 규칙 추가
- [ ] Step 7: 가드레일/에러 처리/에스컬레이션 기준 추가
- [ ] Step 8: 테스트 시나리오 5개 제안
```

## 산출물 규격

반드시 한국어로 아래를 제공합니다.
1) `#` 섹션이 분명한 **완성형 system prompt**(Markdown)  
2) 필요한 경우에만 **가정/미확정 사항**(짧게)

## 핵심 개념

### 작성 시점 placeholder vs 런타임 변수

- `[[...]]`: 프롬프트 작성자가 채워 넣는 작성 시점 placeholder (출시 전 반드시 제거/치환)
- `{{...}}`: 플랫폼이 실제 값으로 치환하는 런타임 변수. 선언부와 본문 모두 `{{v}}`로 통일 (예: `{{고객명}}님 맞으실까요?`)

> 핵심 규칙은 아래 [필수 규칙](#필수-규칙) 참조.

## 최소 입력 질문 (최대 9개)

유저가 모르면 합리적 기본값을 제안합니다. 단, 톤/필러/흐름 같은 **스타일/UX 설정**에 한정합니다. 모델/도구/정책/가격/가능 여부 같은 외부 사실은 기본값을 만들지 말고 확인하거나 `[[...]]`로 남깁니다.

1. **업무**: 무엇을 처리하는 에이전트인가(아웃바운드 세일즈/예약/CS 등)?
2. **컨텍스트**: 회사/서비스, 말해도 되는 것/금지사항
3. **성공 정의**: “완료”는 무엇이며 언제 사람에게 넘기나
4. **도구/액션**: 어떤 도구가 있고 언제 호출하며 파라미터는 무엇인가
5. **데이터 수집**: 이름/전화/이메일/주문번호/예약번호 등
6. **변수 주입(사전 주입 값)**: 통화 시작 전에 주입 가능한 변수가 있나?
   - 예: `{{고객명}}`, `{{고객사}}`, `{{캠페인명}}`, `{{마지막_구매}}`
7. **대화 형태**: 기본 흐름(대표 흐름) + 주요 분기

기본값(권장):
- 말투/톤: 한국어 존댓말(높임 공손: ~실까요/~드릴까요), 짧게
- 필러: `낮음`
- 추가 설정은 유저가 원할 때만 조정

## 선택 질문 (요청 시에만)

유저가 원하거나, 품질/브랜드 요구가 강한 경우에만 물어봅니다.

1) **말투/톤**: 존댓말/반말, 속도, 길이(짧게), 친근함 정도  
2) **필러**: “음…/어…/아…” 같은 필러를 사용할지
   - 옵션: `꺼짐/낮음/보통/높음`
   - 금지 상황도 같이 정하기(예: 고지/숫자/민감정보/정확성 문장)

## 템플릿 기반 초안

템플릿 파일을 복사해 `[[...]]`만 채우고, 이 플레이북의 규칙(변수/필러/정규화/도구/무음 액션/가드레일)을 반영합니다.

- `references/voice-ai-prompt-template.md`
- 템플릿의 `[[style_rules]]`/`[[turn_taking_rules]]`/`[[tool_rules]]`/`[[guardrails_rules]]`에는 이 playbook의 Rules (must)와 관련 섹션 요약을 넣습니다.

## 필수 규칙

- 섹션을 분리합니다(역할/목표/가드레일/도구/흐름). — LLM이 역할별 컨텍스트를 구분해야 hallucination이 줄어듭니다.
- 한 문장에 한 행동만 둡니다(장문/복문 지양). — 음성에서 복문은 사용자가 중간에 끊기 어렵고, LLM이 여러 행동을 한 턴에 몰아서 실행합니다.
- 기본 응답은 1–2문장으로 말합니다. 길어지면 끊어서 진행합니다. — TTS 지연과 사용자 이탈을 방지합니다.
- 선택지가 필요하면 2–3개만 제시합니다(가능하면 2개). — 음성에서 4개 이상은 기억이 어렵습니다.
- 이모지/이모티콘/특수기호(장식용) 사용 금지. — TTS가 읽거나 무시하며 품질이 저하됩니다.
- 메타 발화 금지(프롬프트/정책/도구/내부 처리 설명 금지). — 사용자 신뢰를 깨고 혼란을 줍니다.
- 도구 결과를 추측해서 말하지 않습니다. — 잘못된 정보 전달은 CS 에스컬레이션으로 이어집니다.
- 전환/종료가 필요하면 발화하지 말고 무음 액션만 실행합니다. — 발화와 액션 사이 레이스 컨디션을 방지합니다.

## 런타임 변수 (`{{...}}`)

vox.ai에서는 통화 시작 전에 `{{고객명}}`처럼 변수를 주입해 개인화/컨텍스트를 제공할 수 있습니다.

### 표기 컨벤션

- **Variables 선언부**: `{{고객명}}` — 플랫폼이 치환하는 대상
- **프롬프트 본문(대화 흐름 등)**: `{고객명}` — "여기에 변수 값이 들어감" 표시

### 권장 워크플로우

1) **유저에게 먼저 질문**
- 통화 시작 전에 주입 가능한 값이 있나요? (예: 고객명/고객사/캠페인/리드 출처/마지막 문의 요약)
- 각 값은 “항상 주입됨(필수)”인가요, “가끔 비어있음(옵션)”인가요?
- 값이 비어있을 수 있다면, 어떤 대체 표현이 자연스러운가요?

2) **변수 목록을 확정**
- 3–7개 수준에서 시작(규칙 폭증 방지)
- 변수 없이도 대화가 성립하는지 확인(옵션 변수가 핵심 문장을 깨면 안 됨)

3) **system prompt에 변수 값을 ‘데이터’로 선언**
- 값은 따옴표로 감싸 두면 빈 문자열/미주입 상태를 다루기 쉽습니다.

```md
# Variables
- 고객명: "{{고객명}}" (비어있을 수 있음)
- 고객사: "{{고객사}}" (비어있을 수 있음)
```

4) **미주입/누락을 전제로 fallback 규칙을 명시**
- 값이 비어있거나, 빈 문자열(`""`)로 주입되었거나, `{{고객명}}`처럼 괄호가 그대로 보이면(주입 실패) **이름/회사명 언급을 회피**합니다.
- 대체 표현은 짧고 보편적인 것으로 고정합니다.

권장 대체:
- 고객명 없음 → “고객님”, “성함”
- 고객사 없음 → 문맥상 자연스러울 때만 “회사/귀사”, 아니면 생략

### 패턴

**패턴 A) 인사에서 1회 개인화(권장)**

```md
## 인사
- 고객명이 있으면: "안녕하세요, {고객명}님. 무엇을 도와드릴까요?"
- 고객명이 없으면: "안녕하세요, 고객님. 무엇을 도와드릴까요?" → (필요시) "성함 여쭤봐도 될까요?"
```

**패턴 B) 확인 질문으로 개인화(옵션)**

```md
## 확인
- 고객명이 있으면: "{고객명}님 맞으실까요?"
- 고객명이 없으면: "성함 여쭤봐도 될까요?"
```

### 가드레일

- 개인화는 과하지 않게: 고객명/고객사는 보통 1회면 충분합니다.
- 민감정보는 주입되어도 굳이 발화하지 않습니다(업무에 꼭 필요할 때만 최소로 확인).

## 자연스러움 (필러)

필러는 대화를 자연스럽게 만들지만, 과하면 불확실하게 들립니다.

### 설정

| 옵션 | 설명 |
|------|------|
| `꺼짐` | 필러 사용 안 함 |
| `낮음` | 응답 첫머리에서 1회만 **(기본값)** |
| `보통` | 필요할 때마다, 단 한 응답 1회 |
| `높음` | 더 자연스럽게, 단 한 응답 1회 |

### 허용 필러

"음…", "어…", "아…", "그…"

### 금지 상황

숫자/시간/날짜, 사과/민감 상황, 확정 안내/요약에서는 필러를 쓰지 않습니다.

## 턴테이킹

음성 대화는 텍스트보다 “턴(발화 순서)”이 더 쉽게 꼬입니다.

### 핵심 규칙

- 한 번에 한 질문만 합니다.
- 답이 불명확하면 확인 질문 1개만 합니다(무한 루프 금지).
- 한 응답은 짧게. 한 번에 한 가지.

### 패턴

**패턴 A) 2-Choice 질문 (권장)**

```md
“지금 바로 연결해드릴까요, 아니면 예약을 잡아드릴까요?”
```

**패턴 B) 침묵/무응답 처리**

```md
1) “혹시 들리실까요?”
2) 계속 무응답이면: “지금 통화가 어려우시면 나중에 다시 연락드릴까요?”
```

**패턴 C) 불명확한 답변 처리**

```md
"아, 그러시군요. 혹시 예약 변경이실까요, 아니면 예약 취소이실까요?"
```

## 도구 & 액션

### 전역 규칙

- 도구가 필요한 상황이면, 기억/추측으로 답하지 말고 **도구를 우선 호출**합니다.
- 도구 실패를 전제로 합니다: 문제 인정 → 1회 재시도 → 대안/에스컬레이션. 추측 금지.
- 되돌리기 어려운 작업(예약 확정/결제/취소)은 사용자 확인 후 실행합니다.
- 사용자에게 “도구/함수/프롬프트/정책/내부 시스템” 같은 메타 표현을 하지 않습니다.

### 실패 처리 예시

- “지금 시스템 확인이 잘 안 돼요. 잠시만요.”
- “아직 확인이 어려워서요. 사람 상담으로 연결해드릴까요, 아니면 다시 연락드릴까요?”

## 무음 액션 (전환/종료)

전환(사람 연결)이나 통화 종료 같은 액션은 말로 설명하면 오히려 어색해지거나 레이스 컨디션이 생길 수 있습니다.

### 핵심 규칙

- 전환/종료가 필요하다고 판단하면, 사용자에게 설명하는 발화를 하지 않습니다.
- 대신 해당 액션을 무음으로 실행합니다.
- 단, 액션 실행이 실패했을 때만 짧게 안내하고 대안을 제시합니다.

## 문자 정규화

음성 에이전트는 구조화 데이터를 그대로 읽으면(특수문자/구분자 등) 오해가 생기기 쉽습니다.  
따라서 **발화용(Spoken)** 과 **도구 입력용(Written)** 을 분리합니다.

아래는 “한국어 음성 대화”를 기본 가정한 예시입니다. 제품/도메인 요구사항에 맞게 조정하세요.

### Email

- Spoken example: `"제이 오 에이치 엔 점 스미스 골뱅이 컴퍼니 점 컴"`
- Written example: `"john.smith@company.com"`
- Rules:
  - “골뱅이/앳/at” → `@`
  - “점/닷/dot” → `.`
  - 공백은 제거
  - 알파벳은 가능하면 한 글자씩 확인(“제이”, “오”, “에이치” …)
  - 애매하면 다시 확인 질문 1개로 확정

### Phone

- Spoken example: `"공일공... 일이삼사... 오육칠팔"` 또는 `"010-1234-5678"`
- Written example: `"01012345678"`
- Rules:
  - 숫자는 한 자리씩 말하게 유도(특히 마지막 4자리)
  - 하이픈/공백/구분자는 제거 후 도구에 전달
  - 국가번호가 필요한 시스템이면 국가번호를 확인(예: `+82`)

### Codes / IDs

- Spoken example: `"에이 비 씨 일이삼"` 또는 `"A B C 1 2 3"`
- Written example: `"ABC123"`
- Rules:
  - 영문은 한 글자씩, 숫자는 한 자리씩
  - 띄어쓰기는 제거
  - 시스템이 대소문자를 구분하지 않으면 기본은 대문자로 정규화(필요 시 예외 명시)

### Dates / times

- 발화는 자연스럽게: `"1월 24일 오후 4시 30분"`
- 예약/일정이면 시간대(타임존) 필요 여부를 확인

## 배포 전 체크리스트

배포/운영 전, system prompt를 빠르게 점검할 때 사용합니다.

### 구조

- `#` 섹션이 분명함: 역할/목표/가드레일/도구/흐름
- 각 문장은 “행동”이 있고 모호하지 않음
- 중복/충돌 규칙이 없음
- `[[...]]` placeholder가 남아있지 않음

### 변수

- 선언부는 `{{v}}`, 본문은 `{v}` 표기가 올바르게 구분됨
- “값이 없을 때 / 빈 문자열일 때 / 주입 실패일 때” 세 가지 fallback 규칙이 있음
- 고객명/고객사 같은 변수는 과도하게 반복하지 않음

### 자연스러움

- 필러 사용 여부/강도가 명시됨(꺼짐/낮음/보통/높음) — 기본값은 `낮음`
- 한 응답에 필러 1회 이하
- 금지 상황(고지/숫자/민감정보/정확성 문장)이 명시됨

### 음성

- 짧은 문장, 장문/장황한 설명 없음
- 질문은 한 번에 하나로 유지됨
- 친절하지만 과장/군더더기/기업 말투 과다 없음
- 이모지/이모티콘/장식 문자가 없음

### 도구

- 각 도구에: 언제/어떻게/실패 시 행동이 있음
- 도구 입력이 Written 기준인지 명시됨
- 전환/종료가 있다면 “무음 액션” 규칙이 포함됨

### 신뢰성

- `# 가드레일` 섹션이 따로 있음
- “추측하지 않기 / 사실을 만들어내지 않기”가 포함됨
- 에스컬레이션(사람 연결) 기준이 명확함
- 도구 실패 재시도 정책이 명시됨(1–2회)

### 데이터 수집

- 이메일/전화/코드 수집 시 정규화(Spoken/Written) 규칙이 있음
- 애매한 입력은 확인 질문 1개로 확정하는 절차가 있음

### 테스트

- 최소 5개 시나리오가 있음
- 포함: 도구 실패, 불명확한 사용자, 화난 사용자, 주제 이탈/부적절 요청
