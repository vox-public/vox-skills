# POC Linear 구조 템플릿

새 고객 POC를 시작할 때, 아래 구조대로 Linear에 생성한다.
이 파일은 **Linear 생성 스펙만** 다룬다. 각 단계의 실행 가이드는 `poc-checklist.md`를 참조한다.

## 필수 입력

1. **고객사명**
2. **첫 PoC 데모 전달일 (DueDate)** — 1차 데모를 고객에게 전달하는 목표 날짜
3. **org_id** — 고객사의 vox.ai organization ID (에이전트 생성/수정에 필수)

## 에이전트 이름 규칙

에이전트 이름은 `[회사명] 목적` 형식으로 생성한다.
- 예: `[한독] 바로잰핏 인바운드`, `[닥터다이어리] 코칭 아웃바운드`, `[삼일텍스톤] CS 인바운드`

## 일정 자동 산정

DueDate를 기준으로 후속 일정을 산정한다. 주말(토/일)은 제외한 영업일 기준이다.

- **3단계 "1차 데모 전달"** 이슈의 dueDate = 입력받은 DueDate
- **4단계 "피드백 수집"** 이슈의 dueDate = DueDate + 2영업일
- **4단계 "피드백 반영 수정"** 이슈의 dueDate = DueDate + 4영업일 (2 + 2)

---

## 초기 생성 범위: 1~4단계만

POC 시작 시 **1~4단계(피드백 & 수정)까지만** 이슈를 생성한다.
5~7단계(고객 온보딩, 모니터링, 종료)는 4단계 완료 후 상황에 맞게 생성한다.
이유: POC 초기에는 데모 전달과 1차 피드백까지가 확실한 범위이고, 그 이후는 고객 반응에 따라 진행 방식이 달라질 수 있다.

---

## 이슈 생성 순서 (중요)

Linear에서 이슈는 생성된 순서대로 리스트에 표시된다.
따라서 **작업 순서대로 하나씩** 생성해야 목록이 자연스럽게 정렬된다.

생성 순서:
1. `1. 초기 온보딩` (단계 이슈)
2. `1.1 배경 및 요구사항 정리` (sub-issue, parentId = 1단계)
3. `1.2 전화번호 연동` (sub-issue, parentId = 1단계)
4. `2. 에이전트 설계 및 제작` (단계 이슈)
5. `2.1 데이터 정리 & 시나리오 구성` (sub-issue, parentId = 2단계)
6. `2.2 프롬프트 작성 & 에이전트 테스트` (sub-issue, parentId = 2단계)
7. `3. 1차 데모 전달` (단계 이슈, dueDate = DueDate)
8. `4. 피드백 & 수정` (단계 이슈)
9. `4.1 피드백 수집` (sub-issue, parentId = 4단계, dueDate = DueDate + 2영업일)
10. `4.2 피드백 반영 수정` (sub-issue, parentId = 4단계, dueDate = DueDate + 4영업일)

각 이슈는 **반드시 위 순서대로 하나씩** 생성한다. 병렬 생성하지 않는다.

---

## 모드 A: 프로젝트 기반 생성

프로젝트로 들어오거나 새로 프로젝트를 생성하는 경우.

### 프로젝트 생성 규칙

- **프로젝트 이름**: `[고객사명] AI 전화 에이전트 PoC`
- **팀**: `Customer`
- **우선순위**: High (2)
- **리드**: 담당 FDE (기본: 현재 사용자)
- **시작일**: 프로젝트 생성일

### 구조

```
프로젝트
├── 이슈: "1. 초기 온보딩"
│   ├── sub-issue: "1.1 배경 및 요구사항 정리"
│   └── sub-issue: "1.2 전화번호 연동"
├── 이슈: "2. 에이전트 설계 및 제작"
│   ├── sub-issue: "2.1 데이터 정리 & 시나리오 구성"
│   └── sub-issue: "2.2 프롬프트 작성 & 에이전트 테스트"
├── 이슈: "3. 1차 데모 전달"          ← dueDate = DueDate
└── 이슈: "4. 피드백 & 수정"
    ├── sub-issue: "4.1 피드백 수집"
    └── sub-issue: "4.2 피드백 반영 수정"

(5~7단계는 4단계 완료 후 생성)
```

### Linear 생성 순서

1. `save_project` — 프로젝트 생성
2. 위 "이슈 생성 순서" 섹션의 순서대로 `save_issue`를 **순차적으로** 10회 호출

---

## 모드 B: 이슈 기반 생성

기존 Linear 이슈가 POC 요청인 경우. 해당 이슈를 parent로 사용한다.

### 구조

```
기존 이슈 (parent)
├── sub-issue: "1. 초기 온보딩"
│   ├── sub-issue: "1.1 배경 및 요구사항 정리"
│   └── sub-issue: "1.2 전화번호 연동"
├── sub-issue: "2. 에이전트 설계 및 제작"
│   ├── sub-issue: "2.1 데이터 정리 & 시나리오 구성"
│   └── sub-issue: "2.2 프롬프트 작성 & 에이전트 테스트"
├── sub-issue: "3. 1차 데모 전달"
└── sub-issue: "4. 피드백 & 수정"
    ├── sub-issue: "4.1 피드백 수집"
    └── sub-issue: "4.2 피드백 반영 수정"

(5~7단계는 4단계 완료 후 생성)
```

### Linear 생성 순서

1. `get_issue` — 기존 이슈 정보 확인 (ID, 팀 등)
2. 위 "이슈 생성 순서" 섹션의 순서대로 `save_issue`를 **순차적으로** 10회 호출
   - 1~4단계 이슈의 parentId = 기존 이슈 ID
   - 각 sub-issue의 parentId = 해당 단계 이슈 ID

## 이슈 상세 목록

아래 목록은 모드 A/B 공통이다. 모드에 따라 parent만 달라진다.

### 단계 1: 초기 온보딩

```
이슈 제목: 1. 초기 온보딩
설명: 고객 배경 및 요구사항 정리, 전화번호 연동 (필요시)
```

세부 작업 (sub-issue):
| 번호 | 제목 | 설명 |
|------|------|------|
| 1.1 | 배경 및 요구사항 정리 | 고객사 정보, 담당자, 도입 범위, 미팅 진행 상황, 필요 자료(소개서/매뉴얼/녹음) 수령 현황 등 정리 |
| 1.2 | 전화번호 연동 | PoC용 070번호 발급, 문자기능, 대표번호 연동, 레터링 서비스 등 통신사 레벨 작업 정리. **필요한 경우에만 진행** |

### 단계 2: 에이전트 설계 및 제작

```
이슈 제목: 2. 에이전트 설계 및 제작
설명: 수령 자료 정리, 시나리오 구성, 프롬프트 작성, 에이전트 테스트
```

세부 작업 (sub-issue):
| 번호 | 제목 | 설명 |
|------|------|------|
| 2.1 | 데이터 정리 & 시나리오 구성 | 수령 자료를 Markdown으로 정리 + Mermaid Flowchart로 응대 시나리오 구성 (예외 포함) |
| 2.2 | 프롬프트 작성 & 에이전트 테스트 | 프롬프트 초안 생성 → 전체 구성 완료 → 엔진 설정 → 내부 품질 테스트 (대표 시나리오 5개 이상) |

### 단계 3: 1차 데모 전달

```
이슈 제목: 3. 1차 데모 전달
설명: 완성 에이전트를 고객에게 전달. 테스트 방법 안내 + 2영업일 피드백 요청.
dueDate: 입력받은 DueDate
```

sub-issue 없음 (단일 이슈)

### 단계 4: 피드백 & 수정

```
이슈 제목: 4. 피드백 & 수정
설명: 고객 피드백 1회 수집 및 반영
```

세부 작업 (sub-issue):
| 번호 | 제목 | 설명 | dueDate |
|------|------|------|---------|
| 4.1 | 피드백 수집 | 고객 테스트 후 피드백 수집 및 분류 (프롬프트/설정/범위밖) | **DueDate + 2영업일** |
| 4.2 | 피드백 반영 수정 | 피드백 반영 → 프롬프트/설정 수정 → 재전달 + 에이전트 링크 공유 | **DueDate + 4영업일** |

## 후속 단계 (4단계 완료 후 생성)

4단계가 끝나면 사용자에게 "5~7단계 이슈를 생성할까요?"라고 질문한다.
승인하면 아래 이슈를 **순서대로** 생성한다.

### 단계 5: 고객 온보딩

```
이슈 제목: 5. 고객 온보딩
설명: 온라인 미팅을 통해 다음 항목을 안내한다:
- 통화 정보 추출 방법 (전화번호, 발신자, 목적, 통화 요약)
- 대량발신 (콜 캠페인) 사용법
- 단건 발신 방법
- 웹에서 테스트하는 방법
- 대시보드 사용법 (통화기록 확인, 팀원 공유)
```

### 단계 6: PoC 모니터링

```
이슈 제목: 6. PoC 모니터링
설명: 고객 테스트 일정 확보 후 모니터링 진행.
- 통화 로그 확인 (정상/실패 통화 파악)
- 문제점·개선점 내부 기록
- 성과 지표 정리 (통화 성공률, 평균 통화시간, 에스컬레이션 비율 등)
```

### 단계 7: PoC 종료

```
이슈 제목: 7. PoC 종료
설명: PoC 결과 리포트 정리, 고객에게 도입 여부 확인.
도입 확정 시 본 계약 진행 (세일즈팀 연계).
```

## 영업일 계산 로직

dueDate를 기반으로 영업일을 계산할 때, 주말(토요일=6, 일요일=0)을 건너뛴다.

```
function addBusinessDays(startDate, days):
  current = startDate
  added = 0
  while added < days:
    current = current + 1일
    if current.dayOfWeek != 토 and current.dayOfWeek != 일:
      added += 1
  return current
```

생성 완료 후 프로젝트 URL(모드 A) 또는 parent 이슈 URL(모드 B)을 사용자에게 공유한다.
