# POC 단계별 실행 가이드

각 단계에서 FDE가 무엇을 해야 하는지, 어떤 산출물이 필요한지, 주의할 점은 무엇인지를 정리한다.
Linear 구조 생성 스펙은 `poc-milestones.md`를 참조한다. 이 파일은 **실행 가이드만** 다룬다.

---

## 공통 규칙

아래 규칙은 **모든 단계**에 적용된다. 각 단계 가이드에서 반복하지 않으므로 여기서 숙지한다.

### Linear 업데이트 규칙

각 단계의 작업이 끝나면 사용자에게 **"Linear를 업데이트할까요?"**라고 질문한다.

승인 시:
1. 해당 이슈 상태를 **Done**으로 변경
2. 작업 내용을 이슈 댓글에 기록
3. **에이전트 관련 작업 완료 시** (3단계 데모 전달, 4단계 수정 완료 등):
   - 반드시 댓글로 에이전트 링크 공유: `에이전트 완료: https://www.tryvox.co/agent/{agent_id}`
   - Linear MCP의 `save_comment`를 사용

### 에이전트 이름 규칙

에이전트 이름은 `[회사명] 목적` 형식으로 생성한다.
- 예: `[한독] 바로잰핏 인바운드`, `[닥터다이어리] 코칭 아웃바운드`

### org_id 규칙

에이전트 생성/수정 전에 고객의 vox.ai **org_id**를 반드시 확보한다.
org_id가 있어야 vox MCP로 에이전트를 생성·수정할 수 있다.

## 1단계: 초기 온보딩

> **Skip 조건**: 킥오프 미팅 전에 요구사항, 자료, 일정이 이미 모두 확보되어 있으면 이 단계를 skip하고 2단계로 바로 진행한다.

### 목표
고객사의 배경을 정리하고, 필요한 경우 전화번호 연동 작업을 진행한다.

### 체크리스트

- [ ] 고객사 정보 파악 (회사명, 담당자, 연락처, 도입 범위)
- [ ] 인바운드/아웃바운드 구분 및 처리할 업무 범위 확정
- [ ] 미팅 진행 상황 및 필요 자료 수령 현황 정리
- [ ] 에이전트 설계 자료 수령 확인 (소개서, 카탈로그, 통화녹음, 응대매뉴얼)
- [ ] (필요시) 전화번호 연동 작업 — 070번호, 문자기능, 대표번호, 레터링 서비스 등

### 가이드

**1.1 배경 및 요구사항 정리:**
- 고객사, 담당자, 도입하려는 솔루션 범위(콜센터 규모, IB/OB, 필요 기능 등) 확인
- 킥오프 미팅 내용, 일정 합의 사항, 성공 기준 정리
- 필요 자료 목록: 회사/서비스 소개서, 제품 카탈로그, 통화 녹음 파일, 응대 매뉴얼/FAQ, 자주 들어오는 문의 유형

**1.2 전화번호 연동 (필요시에만):**
- PoC용 070번호 발급 필요 여부 확인
- 문자 기능(SMS) 연동 필요 여부
- 기존 대표번호 연동 또는 레터링 서비스 필요 여부
- 통신사 레벨에서 필요한 작업 목록 정리 및 진행

### 주의사항
- 자료가 늦어지면 2단계 전체 일정이 밀린다. 수령 기한을 명확히 정하자.
- 전화번호 연동은 통신사 처리 시간이 필요하므로 일찍 확인해두자.

## 2단계: 에이전트 설계 및 제작

### 목표
고객사 데이터를 기반으로 에이전트를 설계하고, 내부 테스트를 완료한다.

### 체크리스트

- [ ] **고객 org_id 확보** (공통 규칙 참조)
- [ ] 수령 자료를 Markdown으로 정리
- [ ] 응대 시나리오 Mermaid Flowchart 작성 (예외 포함)
- [ ] 프롬프트 초안 생성 (vox-best-practice 스킬 활용)
- [ ] 전체 프롬프트 구성 완료 + 엔진 설정
- [ ] 내부 품질 테스트 (대표 시나리오 5개 이상)

### 가이드

**자료 정리 → 프롬프트 작성 흐름:**

```
고객 자료 수령
    ↓
Markdown으로 정리 (시나리오별 분류)
    ↓
Mermaid Flowchart 작성 (분기/예외 포함)
    ↓
프롬프트 초안 생성 (vox-best-practice 워크플로우)
    ↓
    ├── single prompt 에이전트 → vox-best-practice 스킬
    └── flow 에이전트 노드 → flow-node-creator 스킬
    ↓
vox MCP로 에이전트 생성 & 프롬프트 적용 (org_id 필요)
    ↓
내부 테스트 (5개 시나리오)
```

**프롬프트 작성 시 확인할 것:**
- vox-best-practice의 8단계 워크플로우를 따른다.
- 런타임 변수(`{{고객명}}` 등)의 미주입 시 대체 표현을 반드시 정의한다.
- 도구(end_call, transfer_call 등) 실패 처리를 포함한다.

**내부 테스트 필수 시나리오:**
1. 정상 흐름 (가장 빈번한 시나리오)
2. 도구 실패 (타임아웃/NotFound)
3. 불명확한 고객 응답
4. 여러 요청 동시 처리
5. 범위 이탈 / 부적절 요청

### 주의사항
- 프롬프트에 `[[...]]` placeholder가 남아있으면 안 된다 (출시 전 모두 제거).
- 테스트 없이 3단계로 넘어가지 않는다.
- DueDate까지 완료해야 하므로 일정을 역산해서 작업한다.

## 3단계: 1차 데모 전달

### 목표
완성된 에이전트를 고객에게 전달하고, 테스트를 요청한다.

### 체크리스트

- [ ] 에이전트 최종 점검 (내부 테스트 통과 확인)
- [ ] 데모 전달물 작성 (아래 템플릿 참조)
- [ ] 고객에게 데모 전달 (테스트 방법 안내)
- [ ] 2영업일 테스트 후 피드백 요청

### 가이드

**데모 전달 시 안내할 것:**
- 전화번호 및 테스트 방법
- 어떤 시나리오를 중심으로 테스트해달라는 가이드
- 피드백 수집 기한 (2영업일 후)
- 피드백 전달 방법 (슬랙/이메일 등)

### 데모 전달물 템플릿

데모 전달물의 형식은 자유다. 상황에 맞는 포맷을 선택한다.

**포맷 옵션:**
- **Mermaid Flowchart** — 시나리오 분기가 복잡할 때. 한눈에 흐름을 볼 수 있어 기술 이해도가 있는 고객에게 적합
- **PDF** — 정돈된 형태로 전달하고 싶을 때. 비기술 의사결정자가 포함된 경우 적합
- **Markdown** — 빠르게 작성하고 슬랙/노션으로 공유할 때. 실무 담당자 간 소통에 적합

**전달물에 반드시 포함할 내용:**

1. **에이전트 개요**: 에이전트 이름, 용도(IB/OB), 처리하는 업무 범위
2. **테스트 방법**: 테스트 전화번호(070), 웹 테스트 링크(해당 시), 테스트 시 유의사항
3. **응대 시나리오 요약**: 에이전트가 처리하는 주요 시나리오 목록 (3~7개), 각 시나리오별 예시 대화 또는 흐름 요약
4. **알려진 제한사항**: 현재 에이전트가 처리하지 못하는 범위, 에스컬레이션(사람 연결) 조건
5. **피드백 요청**: 피드백 기한 (예: "X월 X일까지"), 피드백 전달 방법, 확인해주셨으면 하는 포인트 (예: 첫 인사 자연스러운지, 시나리오 분기 정확한지, 빠진 시나리오 있는지)

**전달 메시지 예시 (슬랙/이메일):**

```
안녕하세요, [담당자님]. vox.ai [FDE이름]입니다.

[고객사명] AI 전화 에이전트 1차 데모를 전달드립니다.

- 테스트 전화번호: 070-XXXX-XXXX
- 에이전트 링크: https://www.tryvox.co/agent/{agent_id}
- 시나리오 요약: [첨부 또는 링크]

테스트해보시고, [피드백 기한]까지 피드백 부탁드립니다.
특히 아래 부분을 중점적으로 확인해주시면 감사하겠습니다:
1. 첫 응대 멘트가 자연스러운지
2. 주요 시나리오 흐름이 맞는지
3. 빠진 시나리오나 추가하고 싶은 기능이 있는지

감사합니다.
```

### 주의사항
- 데모 전달은 DueDate에 맞춘다.
- "피드백 없으면 만족으로 간주"가 아니라, 반드시 피드백을 받아야 한다.

## 4단계: 피드백 & 수정

### 목표
고객 피드백을 반영하여 에이전트를 1회 수정한다.

### 체크리스트

- [ ] 고객 피드백 수집 (DueDate + 2영업일)
- [ ] 피드백 분류 (프롬프트 수정 / 설정 변경 / 범위 밖)
- [ ] 피드백 반영 → 프롬프트/설정 수정 (1-2영업일)
- [ ] 수정된 에이전트 재전달

### 가이드

**피드백 수집 시 확인할 것:**
- 첫 응대 메시지가 자연스러운가?
- 발화 속도/길이가 적절한가?
- 시나리오별 분기가 정확한가?
- 도구(전환/종료) 동작이 올바른가?
- 고객이 원하는데 빠진 시나리오가 있는가?

**피드백 분류:**
- **프롬프트 수정**: 멘트 변경, 시나리오 추가/수정, 톤 조절 → 바로 반영
- **설정 변경**: 발화 속도, 모델 변경 등 → 바로 반영
- **범위 밖**: PoC 범위를 넘는 기능 요청 → 기록해두고, 7단계(PoC 종료)에서 본 계약 논의 시 활용

**수정 시 vox-best-practice 활용:**
- 피드백이 프롬프트 문제라면 진단(diagnosis) → 리팩터링(revision) 워크플로우를 따른다.
- vox MCP의 `update_agent`로 반영한다.

### 주의사항
- PoC 단계이므로 피드백 반영은 1회로 한다.
- 일정: 고객 테스트 2영업일 + 수정 1-2영업일 = 총 3-4영업일

## 5단계: 고객 온보딩

### 목표
고객이 vox.ai 플랫폼을 독립적으로 사용할 수 있도록 온보딩한다.

### 체크리스트

- [ ] 고객에게 연락하여 온보딩 미팅 일정 확정
- [ ] 온보딩 미팅 진행 (30분~1시간)

### 가이드

> **`dashboard-guide` 스킬을 참조한다.** 이 스킬에 에이전트 설정, 테스트, 대량발신, 통화 데이터 조회의 상세 가이드가 있다. `/vox-fde:dashboard-guide` 커맨드로도 실행할 수 있다.

**온보딩 미팅 순서 (dashboard-guide 스킬의 4개 파트):**

1. **에이전트 변경** — 프롬프트 수정, TTS/속도, 툴, 통화정보 추출, 버전 배포
2. **테스트** — 웹 테스트, {{변수}} 프리셋 설정
3. **대량발신** — 프로젝트 생성, 발신 설정, 예약
4. **통화 데이터 조회** — 통화 기록, 추출 칼럼, 필터, 엑셀 다운로드

### 주의사항
- 미팅 전에 고객사 대시보드가 정상 동작하는지 미리 확인한다.
- 화면 공유하면서 직접 보여주는 게 가장 효과적이다.

## 6단계: PoC 모니터링

### 목표
고객의 실 사용을 모니터링하여 성과를 측정한다.

### 체크리스트

- [ ] 고객으로부터 실제 테스트 일정(시작일, 기간) 공유받기
- [ ] 테스트 기간 동안 통화 로그 모니터링
- [ ] 잘 되는 점 / 문제점 / 개선점 내부 기록
- [ ] 성과 지표 정리

### 가이드

**모니터링 방법:**
- vox MCP의 `get_call`로 개별 통화 로그를 확인한다.
- 주기적으로 (매일 또는 격일) 통화 로그를 확인한다.
- 다음을 기록한다:
  - 정상 처리된 통화 수
  - 실패/이탈한 통화 수 및 원인
  - 고객이 보고한 이슈
  - 예상치 못한 시나리오

**성과 측정 지표 (예시):**
- 총 통화 수
- 정상 완료율 (목적 달성 비율)
- 평균 통화 시간
- 에스컬레이션(사람 연결) 비율
- 고객 만족도 (정성 피드백)

### 주의사항
- 모니터링 중 심각한 문제가 발견되면 즉시 프롬프트를 수정한다 (vox-best-practice 진단/리팩터링).
- 성과 데이터는 객관적으로 정리한다. 좋은 점과 개선 필요한 점 모두 포함.

## 7단계: PoC 종료

### 목표
PoC 결과를 정리하고, 고객의 도입 여부를 확인하여 본 계약을 진행한다.

### 체크리스트

- [ ] PoC 결과 리포트 작성 (성과 데이터 기반)
- [ ] 고객에게 도입 여부 확인
- [ ] 도입 확정 시 본 계약 진행 (세일즈팀 연계)

### 가이드

**PoC 결과 리포트:**
- 성과 데이터를 정리하여 고객에게 공유한다.
- "PoC 기간 동안 X건의 통화를 처리했고, 정상 완료율 Y%를 달성했습니다" 형태로 제시.
- 4단계에서 기록해둔 "범위 밖" 요청도 본 계약 논의 시 활용한다.

**도입 확정 시:**
- 본 계약 조건/가격은 세일즈팀과 조율한다 (FDE가 직접 가격 협상을 하지 않는다).
- 세일즈팀에 PoC 결과 리포트 + 고객 피드백을 전달한다.

### 주의사항
- 가격/계약 조건 협상은 FDE 범위가 아니다. 세일즈팀에 넘긴다.
