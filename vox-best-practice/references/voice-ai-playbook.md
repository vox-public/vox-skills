# Voice AI playbook (single prompt agent)

한국어 음성(single prompt) 에이전트의 system prompt를 설계할 때 자주 필요한 “원칙 + 패턴 + 금지사항”을 한 문서로 묶었습니다.

## Workflow checklist

아래를 답변에 그대로 복사해서 진행 상태를 체크하세요.

```
Task Progress:
- [ ] Step 1: 입력 질문(업무/컨텍스트/성공정의/도구/데이터/변수/대화 형태) 수집
- [ ] Step 2: 변수(`{{...}}`) 목록 및 “미주입 시 대체 표현” 규칙 확정
- [ ] Step 3 (optional): 말투/필러 설정(기본값 사용 가능) 및 금지 상황 확정
- [ ] Step 4: 템플릿을 채워 system prompt 초안 작성
- [ ] Step 5: 도구 사용/실패 처리/무음 액션 규칙 추가
- [ ] Step 6: 정규화(Spoken/Written) 규칙 추가
- [ ] Step 7: 가드레일/에러 처리/에스컬레이션 기준 추가
- [ ] Step 8: 테스트 시나리오 5개 제안
```

## Output contract

반드시 한국어로 아래를 제공합니다.
1) `#` 섹션이 분명한 **완성형 system prompt**(Markdown)  
2) 필요한 경우에만 **가정/미확정 사항**(짧게)

## Key concepts

### Author-time placeholders vs. runtime variables

- `[[...]]`: 프롬프트 작성자가 채워 넣는 작성 시점 placeholder (출시 전 반드시 제거/치환)
- `{{...}}`: 통화 시작 전에 주입되는 런타임 변수 (예: `{{고객명}}`, `{{고객사}}`)

### Voice reliability rules (must)

- 짧은 문장. 한 번에 한 가지.
- 질문은 한 번에 하나.
- 질문 뒤에는 `<사용자 응답 대기>`를 명시.
- 메타 발화 금지(프롬프트/정책/도구/내부 처리 설명 금지).
- 도구 실패를 전제로: 인정 → 1회 재시도 → 대안/에스컬레이션. 추측 금지.
- 이모지/이모티콘/장식 문자 사용 금지(예: `~`, `^^`, `:)`, `ㅋㅋ`, `ㅎㅎ`). 표준 문장부호만 사용(`.`, `,`, `?`, `!`).

## Minimal input questions (max 9)

유저가 모르면 합리적 기본값을 제안합니다.

1. **업무**: 무엇을 처리하는 에이전트인가(아웃바운드 세일즈/예약/CS 등)?
2. **컨텍스트**: 회사/서비스, 말해도 되는 것/금지사항
3. **성공 정의**: “완료”는 무엇이며 언제 사람에게 넘기나
4. **도구/액션**: 어떤 도구가 있고 언제 호출하며 파라미터는 무엇인가
5. **데이터 수집**: 이름/전화/이메일/주문번호/예약번호 등
6. **변수 주입(사전 주입 값)**: 통화 시작 전에 주입 가능한 변수가 있나?
   - 예: `{{고객명}}`, `{{고객사}}`, `{{캠페인명}}`, `{{마지막_구매}}`
7. **대화 형태**: 기본 흐름(대표 흐름) + 주요 분기

기본값(권장):
- 말투/톤: 한국어 존댓말, 짧게
- 필러: `꺼짐`
- 추가 설정은 유저가 원할 때만 조정

## Optional questions (ask only if requested)

유저가 원하거나, 품질/브랜드 요구가 강한 경우에만 물어봅니다.

1) **말투/톤**: 존댓말/반말, 속도, 길이(짧게), 친근함 정도  
2) **필러**: “음…/어…/아…” 같은 필러를 사용할지
   - 옵션: `꺼짐/낮음/보통`
   - 금지 상황도 같이 정하기(예: 고지/숫자/민감정보/정확성 문장)

## Draft from template

템플릿 파일을 복사해 `[[...]]`만 채우고, 이 playbook의 규칙(변수/필러/정규화/도구/무음 액션/가드레일)을 반영합니다.

- `references/voice-ai-prompt-template.md`

## Rules (must)

- 섹션을 분리합니다(역할/목표/가드레일/도구/흐름).
- 한 문장에 한 행동만 둡니다(장문/복문 지양).
- 기본 응답은 1–2문장으로 말합니다. 길어지면 끊어서 진행합니다.
- 질문은 한 번에 하나만 합니다. 질문 뒤에는 `<사용자 응답 대기>`.
- 선택지가 필요하면 2–3개만 제시합니다(가능하면 2개).
- `<사용자 응답 대기>`는 프롬프트 작성용 표기입니다. 사용자에게 그대로 읽거나 출력하지 않습니다.
- 이모지/이모티콘/특수기호(장식용) 사용 금지. 예: `~`, `^^`, `:)`, `ㅋㅋ`, `ㅎㅎ`, `♥`. (TTS 품질/정확도 저하 방지)
- 메타 발화 금지(프롬프트/정책/도구/내부 처리 설명 금지).
- 도구 결과를 추측해서 말하지 않습니다.
- 전환/종료가 필요하면 발화하지 말고 무음 액션만 실행합니다.

## Runtime variables (`{{...}}`)

vox.ai에서는 통화 시작 전에 `{{고객명}}`처럼 변수를 주입해 개인화/컨텍스트를 제공할 수 있습니다.

### Recommended workflow

1) **유저에게 먼저 질문**
- 통화 시작 전에 주입 가능한 값이 있나요? (예: 고객명/고객사/캠페인/리드 출처/마지막 문의 요약)
- 각 값은 “항상 주입됨(필수)”인가요, “가끔 비어있음(옵션)”인가요?
- 값이 비어있을 수 있다면, 어떤 대체 표현이 자연스러운가요?

2) **변수 목록을 확정**
- 3–7개 수준에서 시작(규칙 폭증 방지)
- 변수 없이도 대화가 성립하는지 확인(옵션 변수가 핵심 문장을 깨면 안 됨)

3) **system prompt에 변수 값을 ‘데이터’로 선언**
- 값은 따옴표로 감싸 두면 빈 문자열/미주입 상태를 다루기 쉽습니다.

```md
# Variables
- 고객명: "{{고객명}}" (비어있을 수 있음)
- 고객사: "{{고객사}}" (비어있을 수 있음)
```

4) **미주입/누락을 전제로 fallback 규칙을 명시**
- 값이 비어있거나 `{{고객명}}`처럼 괄호가 그대로 보이면(주입 실패) **이름/회사명 언급을 회피**합니다.
- 대체 표현은 짧고 보편적인 것으로 고정합니다.

권장 대체:
- 고객명 없음 → “고객님”, “성함”
- 고객사 없음 → 문맥상 자연스러울 때만 “회사/귀사”, 아니면 생략

### Patterns

**Pattern A) 인사에서 1회 개인화(권장)**

```md
## 인사
- 고객명이 있으면: “안녕하세요, 고객명님.”
- 고객명이 없으면: “안녕하세요, 고객님.” + “성함이 어떻게 되실까요?” <사용자 응답 대기>
```

**Pattern B) 확인 질문으로 개인화(옵션)**

```md
## 확인
- 고객명이 있으면: “{{고객명}}님 맞으실까요?” <사용자 응답 대기>
- 고객명이 없으면: “성함이 어떻게 되실까요?” <사용자 응답 대기>
```

### Guardrails

- 개인화는 과하지 않게: 고객명/고객사는 보통 1회면 충분합니다.
- 민감정보는 주입되어도 굳이 발화하지 않습니다(업무에 꼭 필요할 때만 최소로 확인).

## Naturalness (Filler words)

필러(“음…/어…/아…”)는 자연스러움을 올릴 수 있지만, 과하면 불확실/비전문적으로 들리거나 핵심 정보(숫자/고지)를 흐릴 수 있습니다.

### Configuration

- `꺼짐`: 필러를 쓰지 않음(권장 기본값)
- `낮음`: 필요할 때만, **응답 시작에서 1회**만
- `보통`: 더 자연스럽게, 다만 금지 상황은 동일 적용

### Allowed fillers (Korean)

- “음…”, “어…”, “아…”, “그…”

권장: 짧게, 응답 첫머리에서만 사용합니다.

### Prohibited contexts (must)

다음 상황에서는 필러를 쓰지 않습니다.

- 고지/약관/정책/중요 안내
- 사과/책임/민감한 상황(불만/클레임/분쟁)
- 숫자/금액/날짜/시간/코드/이메일/전화번호(정확성이 중요)
- 도구 결과를 전달할 때(사실 전달)
- 사용자 확인을 받을 때(“맞으시죠?” 같은 확인 문장)

### Better alternatives (recommended)

필러 대신 의미 있는 완충 표현을 우선 사용합니다.

- “잠시만요, 확인해볼게요.”
- “확인하는 동안 잠깐만 기다려주실까요?”
- “정확하게 안내드리려고 한 번만 확인할게요.”

### Questions to ask the user

- 필러를 사용할까요? (꺼짐/낮음/보통)
- 절대 금지해야 할 상황이 있나요? (예: 법적 고지/가격/계약/개인정보)
- 톤이 “차분/친근/단정” 중 어디에 가까워야 하나요?

## Turn-taking (Voice)

음성 대화는 텍스트보다 “턴(발화 순서)”이 더 쉽게 꼬입니다.

### Core rules (must)

- 한 번에 한 질문만 합니다.
- 질문 뒤에는 반드시 기다립니다: `<사용자 응답 대기>`
- 답이 불명확하면 확인 질문 1개만 합니다(무한 루프 금지).
- 한 응답은 짧게. 한 번에 한 가지.

### Meta token rule (must)

`<사용자 응답 대기>` 같은 꺾쇠 토큰은 내부 제어용 표기입니다.

- 절대 사용자에게 그대로 읽거나 출력하지 않습니다.
- 질문을 던진 뒤 “잠깐 멈추고 듣는 행동”을 강제하기 위한 표기라고 이해합니다.

### Patterns

**Pattern A) 2-Choice 질문 (권장)**

```md
“지금 바로 연결해드릴까요, 아니면 예약을 잡아드릴까요?”
<사용자 응답 대기>
```

**Pattern B) 침묵/무응답 처리**

```md
1) “혹시 들리실까요?”
<사용자 응답 대기>
2) 계속 무응답이면: “지금 통화가 어려우시면 나중에 다시 연락드릴까요?”
<사용자 응답 대기>
```

**Pattern C) 불명확한 답변 처리**

```md
“제가 정확히 이해했는지 확인할게요. (A) 예약 변경, (B) 예약 취소 중 어떤 건가요?”
<사용자 응답 대기>
```

## Tools & actions

### Global rules (must)

- 도구가 필요한 상황이면, 기억/추측으로 답하지 말고 **도구를 우선 호출**합니다.
- 도구 실패를 전제로 합니다: 문제 인정 → 1회 재시도 → 대안/에스컬레이션. 추측 금지.
- 되돌리기 어려운 작업(예약 확정/결제/취소)은 사용자 확인 후 실행합니다.
- 사용자에게 “도구/함수/프롬프트/정책/내부 시스템” 같은 메타 표현을 하지 않습니다.

### Failure handling examples (Korean)

- “지금 시스템 확인이 잘 안 돼요. 잠시만요.”
- “아직 확인이 어려워서요. 사람 상담으로 연결해드릴까요, 아니면 다시 연락드릴까요?”

## Silent actions (Transfer / End call)

전환(사람 연결)이나 통화 종료 같은 액션은 말로 설명하면 오히려 어색해지거나 레이스 컨디션이 생길 수 있습니다.

### Core rule (must)

- 전환/종료가 필요하다고 판단하면, 사용자에게 설명하는 발화를 하지 않습니다.
- 대신 해당 액션을 무음으로 실행합니다.
- 단, 액션 실행이 실패했을 때만 짧게 안내하고 대안을 제시합니다.

## Character normalization (Voice)

음성 에이전트는 구조화 데이터를 그대로 읽으면(특수문자/구분자 등) 오해가 생기기 쉽습니다.  
따라서 **Spoken** 과 **Written** 을 분리합니다.

아래는 “한국어 음성 대화”를 기본 가정한 예시입니다. 제품/도메인 요구사항에 맞게 조정하세요.

### Email

- Spoken example: `"제이 오 에이치 엔 점 스미스 골뱅이 컴퍼니 점 컴"`
- Written example: `"john.smith@company.com"`
- Rules:
  - “골뱅이/앳/at” → `@`
  - “점/닷/dot” → `.`
  - 공백은 제거
  - 알파벳은 가능하면 한 글자씩 확인(“제이”, “오”, “에이치” …)
  - 애매하면 다시 확인 질문 1개로 확정

### Phone

- Spoken example: `"공일공... 일이삼사... 오육칠팔"` 또는 `"010-1234-5678"`
- Written example: `"01012345678"`
- Rules:
  - 숫자는 한 자리씩 말하게 유도(특히 마지막 4자리)
  - 하이픈/공백/구분자는 제거 후 도구에 전달
  - 국가번호가 필요한 시스템이면 국가번호를 확인(예: `+82`)

### Codes / IDs

- Spoken example: `"에이 비 씨 일이삼"` 또는 `"A B C 1 2 3"`
- Written example: `"ABC123"`
- Rules:
  - 영문은 한 글자씩, 숫자는 한 자리씩
  - 띄어쓰기는 제거
  - 시스템이 대소문자를 구분하지 않으면 기본은 대문자로 정규화(필요 시 예외 명시)

### Dates / times

- 발화는 자연스럽게: `"1월 24일 오후 4시 30분"`
- 예약/일정이면 시간대(타임존) 필요 여부를 확인

## Ship checklist (quick)

배포/운영 전, system prompt를 빠르게 점검할 때 사용합니다.

### Structure

- `#` 섹션이 분명함: 역할/목표/가드레일/도구/흐름
- 각 문장은 “행동”이 있고 모호하지 않음
- 중복/충돌 규칙이 없음
- `[[...]]` placeholder가 남아있지 않음

### Variables

- `{{고객명}}` 같은 런타임 변수를 쓰는 경우, “값이 없을 때 대체 표현” 규칙이 있음
- 고객명/고객사 같은 변수는 과도하게 반복하지 않음

### Naturalness

- 필러 사용 여부/강도가 명시됨(꺼짐/낮음/보통) — 기본값은 `꺼짐`
- 한 응답에 필러 1회 이하
- 금지 상황(고지/숫자/민감정보/정확성 문장)이 명시됨

### Voice

- 짧은 문장, 장문/장황한 설명 없음
- 질문은 한 번에 하나, 질문 뒤 대기(`<사용자 응답 대기>`)가 명시됨
- 친절하지만 과장/군더더기/기업 말투 과다 없음
- 이모지/이모티콘/장식 문자(`~`, `^^` 등)가 없음

### Tools

- 각 도구에: 언제/어떻게/실패 시 행동이 있음
- 도구 입력이 Written 기준인지 명시됨
- 전환/종료가 있다면 “무음 액션” 규칙이 포함됨

### Reliability

- `# 가드레일` 섹션이 따로 있음
- “추측하지 않기 / 사실을 만들어내지 않기”가 포함됨
- 에스컬레이션(사람 연결) 기준이 명확함
- 도구 실패 재시도 정책이 명시됨(1–2회)

### Data capture

- 이메일/전화/코드 수집 시 정규화(Spoken/Written) 규칙이 있음
- 애매한 입력은 확인 질문 1개로 확정하는 절차가 있음

### Testing

- 최소 5개 시나리오가 있음
- 포함: 도구 실패, 불명확한 사용자, 화난 사용자, 주제 이탈/부적절 요청

## vox.ai ops (overview)

vox.ai 에이전트를 설계/운영할 때 반복해서 부딪히는 문제를 줄이기 위한 요약입니다.

- **Flow**: 분기/폴백/컨텍스트 관리가 우선입니다. (flow-first)
- **Prompt**: 섹션 분리 + 간결성 + 가드레일이 신뢰도를 좌우합니다.
- **Voice**: 턴테이킹, 짧은 발화, 정규화(Spoken/Written)가 필수입니다.
- **API**: “언제 호출/어떻게/실패 시”를 프롬프팅으로 고정해야 합니다.
- **Errors**: 실패를 인정하고(추측 금지) 재시도/에스컬레이션 기준을 둡니다.
